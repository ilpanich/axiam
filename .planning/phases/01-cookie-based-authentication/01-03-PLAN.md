**Cookie jar approach in actix-web tests:**
Actix-web test utilities don't have a built-in cookie jar. Use manual cookie extraction and forwarding:

```rust
// Helper: extract a named cookie from response Set-Cookie headers
fn extract_cookie<'a>(resp: &'a actix_web::test::ServiceResponse, name: &str) -> Option<String> {
    resp.response()
        .cookies()
        .find(|c| c.name() == name)
        .map(|c| c.value().to_owned())
}

// Helper: extract cookie attributes for verification
fn extract_cookie_obj<'a>(resp: &'a actix_web::test::ServiceResponse, name: &str) -> Option<actix_web::cookie::Cookie<'a>> {
    // Parse from Set-Cookie header to get attributes
    resp.response().cookies().find(|c| c.name() == name)
}

// Helper: build request with cookies (simulating browser cookie jar)
fn with_cookies(req: actix_web::test::TestRequest, cookies: &[(&str, &str)]) -> actix_web::test::TestRequest {
    let cookie_header = cookies
        .iter()
        .map(|(k, v)| format!("{}={}", k, v))
        .collect::<Vec<_>>()
        .join("; ");
    req.insert_header(("Cookie", cookie_header))
}
```

**Test cases to implement (matching 01-VALIDATION.md task IDs):**

**1. `login_sets_httponly_access_cookie` (01-01-01):**
- POST /api/v1/auth/login with valid credentials
- Assert response status 200
- Extract `axiam_access` cookie from Set-Cookie headers
- Verify cookie exists and is non-empty
- Verify `httpOnly` flag is true
- Verify `SameSite` is `Strict`
- Verify `Path` is `/`
- Verify response body contains `user.id`, `user.username`, `user.email`, `session_id`, `expires_in`
- Verify response body does NOT contain `access_token` or `refresh_token` keys

**2. `login_sets_pathscoped_refresh_cookie` (01-01-02):**
- Same login call as above
- Extract `axiam_refresh` cookie
- Verify cookie exists and is non-empty
- Verify `httpOnly` is true
- Verify `Path` is `/api/v1/auth/refresh`
- Verify `SameSite` is `Strict`

**3. `login_sets_csrf_cookie`:**
- Same login call
- Extract `axiam_csrf` cookie
- Verify cookie exists and is non-empty
- Verify `httpOnly` is false (JS-readable)
- Verify `Path` is `/`

**4. `csrf_missing_header_returns_403` (01-01-03):**
- Login to get cookies
- POST to a state-changing endpoint (e.g., POST /api/v1/auth/logout) with `axiam_access` cookie but WITHOUT `X-CSRF-Token` header
- Assert response status 403

**5. `csrf_valid_header_allows_request`:**
- Login to get cookies (axiam_access, axiam_csrf)
- POST to a state-changing endpoint with `axiam_access` cookie AND `X-CSRF-Token` header matching the `axiam_csrf` cookie value
- Assert request is NOT rejected for CSRF (may get other errors, but not 403 CSRF)

**6. `csrf_get_request_passes_without_token`:**
- Login to get axiam_access cookie
- GET /api/v1/auth/me with only `axiam_access` cookie (no CSRF token)
- Assert response status 200 (GET is exempt from CSRF)

**7. `logout_clears_cookies` (01-01-04):**
- Login to get all three cookies
- POST /api/v1/auth/logout with cookies + CSRF header
- Assert response status 204
- Verify all three cookies in Set-Cookie have `Max-Age=0` (removal cookies)

**8. `refresh_uses_cookie_returns_new_access_cookie` (01-01-05):**
- Login to get cookies
- POST /api/v1/auth/refresh with `axiam_refresh` cookie (and CSRF header if needed — refresh may be CSRF exempt, check middleware exempt list)
- Assert response status 200
- Verify new `axiam_access` cookie in Set-Cookie
- Verify response body contains `expires_in` but NOT `access_token`

**9. `me_returns_user_info`:**
- Login to get cookies
- GET /api/v1/auth/me with `axiam_access` cookie
- Assert response status 200
- Verify response body contains `user.id`, `user.username`, `user.email`

**10. `me_returns_401_without_cookie`:**
- GET /api/v1/auth/me with no cookies and no Authorization header
- Assert response status 401

**11. Update existing MFA flow tests:**
- `mfa_setup_full_flow_returns_tokens` → rename to `mfa_setup_full_flow_sets_cookies`
- MFA verify response should set cookies, not return tokens in body
- Verify cookies are set after MFA verify
- Use cookie jar pattern throughout

  Choose the pragmatic approach — the goal is verifying the cookie IS set, attributes are correct.
