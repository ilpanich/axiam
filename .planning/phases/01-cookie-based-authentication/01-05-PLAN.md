# Plan 01-05 — Login contract drift gap closure

## Context

Discovered during Phase 2 UAT driving (Playwright against /login):

- `frontend/src/pages/LoginPage.tsx:~post /api/v1/auth/login` sends
  `{ username, password, tenant_slug, org_slug }`
- `crates/axiam-api-rest/src/handlers/auth.rs::LoginRequest` expects
  `{ username_or_email, password, tenant_id: Uuid, org_id: Uuid }`

Result: every UI-driven login returns HTTP 400 `missing field username_or_email` even with
valid credentials. The admin UI has never been able to log in end-to-end. This was NOT
visible from Phase 1 UAT Test 5 protocol probes because those used curl with the backend
schema directly.

01-04 closed the path-scope portion of the gap (request reaches the handler). This plan
closes the remaining body-schema portion: make the backend accept the frontend's actual
payload shape.

## Objective

Backend accepts either the ID-based or the slug-based form of the login body, resolving
slugs server-side. No frontend change required.

## Design

### LoginRequest (new shape)

```rust
#[derive(Debug, Deserialize, utoipa::ToSchema)]
pub struct LoginRequest {
    #[serde(default)]
    pub tenant_id: Option<Uuid>,
    #[serde(default)]
    pub org_id: Option<Uuid>,
    #[serde(default)]
    pub tenant_slug: Option<String>,
    #[serde(default)]
    pub org_slug: Option<String>,
    #[serde(alias = "username")]
    pub username_or_email: String,
    pub password: String,
}
```

### Resolution in the login handler

Before building `LoginInput`, resolve missing IDs from slugs:

```rust
let org_id = match (b.org_id, b.org_slug.as_deref()) {
    (Some(id), _) => id,
    (None, Some(slug)) => org_repo.get_by_slug(slug).await
        .map_err(|_| AxiamApiError(AxiamError::auth::InvalidCredentials))?.id,
    (None, None) => return Err(AxiamApiError(AxiamError::ValidationError {
        field: "org_id_or_slug".into(),
        message: "must provide org_id or org_slug".into(),
    })),
};
let tenant_id = match (b.tenant_id, b.tenant_slug.as_deref()) {
    (Some(id), _) => id,
    (None, Some(slug)) => tenant_repo.get_by_slug(org_id, slug).await
        .map_err(|_| AxiamApiError(AxiamError::auth::InvalidCredentials))?.id,
    (None, None) => return Err(AxiamApiError(AxiamError::ValidationError {
        field: "tenant_id_or_slug".into(),
        message: "must provide tenant_id or tenant_slug".into(),
    })),
};
```

**Security note:** slug-resolution failures are intentionally mapped to
`InvalidCredentials` (not 404) to avoid disclosing whether an organization or tenant with
that slug exists — consistent with the existing "auth never distinguishes username-exists
from password-wrong" pattern.

### Handler signature

Add an `org_repo` injection:
```rust
pub async fn login<C: Connection>(
    req: HttpRequest,
    svc: web::Data<AuthSvc<C>>,
    mfa_svc: web::Data<MfaMethodSvc<C>>,
    settings_repo: web::Data<SurrealSettingsRepository<C>>,
    user_repo: web::Data<SurrealUserRepository<C>>,
    org_repo: web::Data<SurrealOrganizationRepository<C>>,   // NEW
    tenant_repo: web::Data<SurrealTenantRepository<C>>,      // NEW
    auth_config: web::Data<AuthConfig>,
    body: web::Json<LoginRequest>,
)
```

`SurrealOrganizationRepository<C>` is already registered as `web::Data` in
`crates/axiam-server/src/main.rs:147`, so no bootstrapping change is needed.
