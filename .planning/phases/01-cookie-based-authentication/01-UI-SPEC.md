# Phase 1 — UI Design Contract: Cookie-Based Authentication

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

Phase 1 is a security refactor, not a feature addition. The user-facing UI (login form, MFA step, layout) already exists and is not being redesigned. This contract:

1. Locks the existing visual design so the refactor does not accidentally drift from it
2. Defines the **interaction state changes** introduced by the cookie migration (auth rehydration on boot, CSRF token injection, silent refresh behavior)
3. Defines copywriting for error conditions that change due to the new auth mechanism

No new screens are added in this phase.

## Interaction States

These states are **changed by the cookie migration** and must be implemented correctly:

### App Initialization (auth rehydration)

**Current behavior (sessionStorage):** App boots → Zustand rehydrates from sessionStorage → user is immediately authenticated.

**New behavior (cookie auth, D-11, D-12):** App boots → Zustand store is memory-only (no persistence) → `GET /api/v1/auth/me` is called → on 200 populate store → on 401 redirect to `/login`.

**Visual contract:**

| State | What the user sees |
|-------|--------------------|
| Initializing (awaiting `/me` response) | Full-page loading state: centered `Loader2` spinner (24px, `animate-spin`) on `bg-axiam-gradient`, no content. Duration: until `/me` resolves. |
| Authenticated | Route proceeds normally. No flash. |
| Unauthenticated (401 from `/me`) | Immediate redirect to `/login`. No error shown — unauthenticated is the expected state. |
| Network error during `/me` | Treat as unauthenticated. Redirect to `/login`. |

### Login Flow (3 steps — unchanged visually)

Step sequence and visual design unchanged from existing `LoginPage.tsx`. The refactor only changes:

- Response parsing: stop reading `access_token` from response body; read `user`, `session_id`, `expires_in` instead (D-10, D-13)
- Store update: call updated `setUser()` instead of `setTokens(accessToken, user)` — no token stored in JS memory
- Zustand store: remove `accessToken` field, `setTokens()` becomes `setUser()` — state shape changes internally but visually identical

### Silent Token Refresh (interceptor)

**Current behavior:** 401 → call `/auth/refresh` with body `{}` → get `access_token` in response → update store → retry.

**New behavior (D-14, D-15):** 401 → call `/auth/refresh` with `withCredentials: true` (cookie carries refresh token automatically) → get `{ expires_in }` in response (no token in body) → no store update needed → retry original request (cookies now refreshed).

**Visual contract:** No UI change. The retry is silent. On refresh failure, existing redirect to `/login` behavior is preserved.

### CSRF Token Injection

**Not visible to the user.** The Axios request interceptor reads `axiam_csrf` cookie via `document.cookie` and injects it as `X-CSRF-Token` header on all state-changing requests (POST, PUT, PATCH, DELETE). GET requests are unaffected (D-03, D-04).

**Error state if CSRF token missing:** The backend will return `403 Forbidden`. The frontend should surface this as a generic error (see Copywriting Contract below).

### Logout

**Current behavior:** Clear store, redirect to `/login`.

**New behavior:** POST `/auth/logout` with `withCredentials: true` → server clears cookies (`Max-Age=0`) → frontend clears Zustand store → redirect to `/login`.

**Visual contract:** No UI change. The logout button/action already exists. The only change is the API call now uses cookies, not Authorization header.

| Condition | Copy |
|-----------|------|

### Error copy — new (Phase 1 introduces)

| Condition | Copy |
|-----------|------|
| CSRF validation failure (403 on state-changing request) | "Request rejected for security reasons. Please refresh the page and try again." |
| Session expired (refresh cookie expired, 401 on `/auth/refresh`) | No inline error shown — redirect to `/login` silently. |
| `/me` endpoint unavailable on init (network error) | No inline error — redirect to `/login` silently. Treat as unauthenticated. |
| Login succeeds but `/me` returns unexpected shape | "Authentication error. Please sign in again." — then redirect to `/login`. |

### Empty state

Not applicable. Phase 1 has no list views or data-driven pages. The login page is always shown when unauthenticated — it is its own empty state.

### Destructive actions

No destructive actions introduced in Phase 1. Logout is a session-clearing action but does not destroy data. No confirmation dialog required.

## Accessibility Contract

Existing login form already implements:

- `role="alert"` on error banner
- `noValidate` on forms (custom validation)
- `aria-hidden="true"` on decorative icons and rings
- `autoFocus` on first field of each step
- `autoComplete` attributes on all credential fields
- `inputMode="numeric"` + `pattern="[0-9]{6}"` on TOTP field

Phase 1 must preserve all existing accessibility attributes during the refactor. The auth initializer loading state must include `aria-live="polite"` or equivalent for screen reader announcement if loading takes more than 500ms.
