# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> a "Locked" badge variant, a locked-user filter, and an "Unlock" action button. All backend
> security header and rate limiting work has no UI surface. This contract governs the lockout
> admin UI only (REQ-3 AC item: "Lockout status visible in admin UI", D-08, D-09).

**New warning token:** The existing design system has no warning/amber token. Add `warning` to the Tailwind config:

```js

warning: {
  DEFAULT: "#f59e0b",
  foreground: "#0d0d2b",
},
```

**Focal point — Users page:** Primary visual anchor is the amber "Locked" badge in the Status column — draws admin attention to accounts requiring action.

## Interaction Contract

### Locked Badge (D-08)

- Position: Status column of the user list table, alongside or replacing the existing `StatusBadge`
- Appearance: `inline-flex items-center px-2 py-0.5 rounded text-xs font-medium bg-amber-500/15 text-amber-400 border border-amber-500/30`
- Label: "Locked"
- When shown: `user.status === "Locked"` OR `user.locked_until !== null`
- Tooltip (optional): Display lockout expiry if `locked_until` is set — `aria-label="Locked until {datetime}"`

### Locked Filter (D-08)

- Position: Filter row between PageHeader and DataTable, alongside the existing SearchInput
- Appearance: A `Button variant="ghost" size="sm"` toggle — active state uses `bg-amber-500/15 text-amber-400 border border-amber-500/30`
- Label: "Locked" with a `Lock` lucide icon (size 14)
- Behavior: Toggles `lockedOnly` state boolean; when true, filters `users` array to those with `status === "Locked"` or `locked_until !== null`; resets to page 1 on toggle
- Active state copy: "Locked (N)" where N is count of locked users

### Unlock Button (D-09)

- Position: Actions column, rightmost — shown only when user is locked
- Appearance: Icon button `p-1.5 rounded hover:bg-cyan-500/10 text-muted-foreground hover:text-cyan-400 transition-colors` with `LockOpen` lucide icon (size 14)
  - Note: `p-1.5` (6px) is an inherited convention matching all other action icon buttons in this table — not derived from the new spacing scale.
- `aria-label`: `"Unlock {username}"`
- On click: Opens `ConfirmDialog` (reuse existing component)
- Confirm dialog title: "Unlock Account"
- Confirm dialog description: "Unlock {username}'s account? They will be able to log in immediately."
- Confirm button label: "Unlock Account"
- Confirm button variant: default (cyan — positive action, not destructive)
- On confirm: Calls `POST /api/v1/users/{id}/unlock` — invalidates `["users"]` query on success
- Success: No toast needed — table re-renders with badge gone (optimistic or refetch)
- Error: Inline error in confirm dialog — `"Failed to unlock account. Please try again."`

## Copywriting Contract

| Element | Copy | Source |
|---------|------|--------|
| Locked badge label | "Locked" | D-08; matches existing "Active" / "Disabled" / "Revoked" capitalization pattern |
| Filter button label (inactive) | "Locked" | Consistent with search row conventions |
| Filter button label (active) | "Locked (N)" | N = count of locked users in current result |
| Unlock button aria-label | "Unlock {username}" | Matches existing "Edit {username}", "Delete {username}" pattern |
| Unlock confirm title | "Unlock Account" | Action-first, no punctuation |
| Unlock confirm body | "Unlock {username}'s account? They will be able to log in immediately." | Confirms consequence; no alarming language (unlock is not destructive) |
| Unlock confirm CTA | "Unlock Account" | Verb + noun — fully self-contained, no "Submit" or "Confirm" |
| Empty state (locked filter active, no locked users) | "No locked accounts." | Follows existing `emptyMessage="No users found."` pattern |
| Error state (unlock fails) | "Failed to unlock account. Please try again." | Problem + next step; consistent with existing error messages in UsersPage |
| Rate limit error (frontend receives 429) | "Too many attempts. Please wait {N} seconds before trying again." | Surfaced on login page — D-03 specifies `retry_after: N` in 429 response body |
| Account lockout error (login blocked) | "Account temporarily locked. Too many failed login attempts. Try again after {time} or contact your administrator." | Displayed on login form when backend returns lockout error |

## States and Interactions Summary

| State | Trigger | Visual Treatment |
|-------|---------|-----------------|
| User is locked | `locked_until !== null` | Amber "Locked" badge in Status column; LockOpen icon in Actions |
| User is active | `is_active === true` | Existing cyan "Active" badge (unchanged) |
| Locked filter on | Filter button toggled | Button gets amber active style; table shows only locked users |
| Locked filter on, zero results | Filter on + no locked users | DataTable shows "No locked accounts." |
| Unlock in progress | `unlockMutation.isPending` | ConfirmDialog submit button disabled + loading spinner |
| Unlock success | Mutation settles | Dialog closes; table re-fetches; badge disappears |
| Unlock error | Mutation rejects | Error text below confirm button in dialog |
| Login rate limited (429) | Frontend receives 429 on login | Inline error on login form with retry_after countdown |
| Login account locked | Backend returns lockout error | Inline error on login form with locked_until time |
