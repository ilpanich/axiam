## New Hook Contract: `usePermissions`

A minimal client-side hook consumed only by `Sidebar.tsx`.

```
usePermissions(): {
  can: (permission: string) => boolean
  permissions: string[]
  isLoading: boolean
}
```

Implementation rules:
- Read permissions from `useAuthStore(s => s.user?.permissions ?? [])`
- `can(permission)` returns `true` if `permissions.includes(permission)` OR if `permissions.includes('*')` (super-admin wildcard)
- No network call — permissions are populated at login time and stored in the auth store
- `isLoading` reflects `useAuthStore(s => s.isInitializing)`

---

## Auth Store Extension

Add `permissions` to the existing `AuthUser` interface in `src/stores/auth.ts`:

```typescript
export interface AuthUser {
  id: string;
  username: string;
  email: string;
  permissions: string[];  // Added in Phase 3
}
```

Permissions are populated from the `/auth/me` or `/oauth2/userinfo` API response when the user logs in. If the API does not yet return permissions in this phase, default to an empty array and the sidebar shows all items (fallback-open, never broken).

| Sidebar Section | Required Permission |
|-----------------|---------------------|
| Dashboard | (always visible — no permission required) |

| Groups | `groups:list` |
| Roles | `roles:list` |
| Permissions | `permissions:list` |
| Resources | `resources:list` |
| Service Accounts | `service_accounts:list` |
| Federation | `federation:list` |
| Organizations | `organizations:list` |
| Tenants | `tenants:list` |
| Certificates | `certificates:list` |
| PGP Keys | `pgp_keys:list` |
| Webhooks | `webhooks:list` |
| OAuth2 Clients | `oauth2_clients:list` |

- Audit Logs and Profile are always visible (self-service access pattern from D-13/D-14)

## Bootstrap Page UI Contract

The bootstrap flow (`POST /api/v1/admin/bootstrap`) requires a minimal public-facing form. This is a one-time setup screen accessed before any admin exists.

Layout: `PublicLayout` wrapper (same as LoginPage). Single centered card, max-width 400px.

Form fields:
- Email — `<input type="email">`, label "Email address", placeholder "admin@example.com"
- Username — `<input type="text">`, label "Username", placeholder "admin"
- Password — `<input type="password">`, label "Password", placeholder "••••••••"

Button: "Create Admin Account" — full-width, `btn-primary` style.

Error states:
- Environment email mismatch → inline error below email field: "This email address is not authorized for bootstrap."
- Already initialized → replace form with message: "This instance has already been initialized." + link to login
- Generic error → inline below form: "Could not create admin account. Verify the server is running and check the server logs."

After success: redirect to `/login` with query param `?bootstrapped=1`. LoginPage displays a toast/notice: "Admin account created. Sign in to continue."

## Accessibility Contract

- All interactive elements reachable via keyboard (Tab order follows DOM order)
- Focus ring visible on all interactive elements (`focus:ring-2 focus:ring-primary/40`)
- Bootstrap form: all inputs have `<label>` with matching `htmlFor`/`id`
- Permission-denied nav items: `aria-disabled="true"` on the `<NavLink>`, `tabIndex={-1}` to remove from tab order
- Loading states: `aria-busy="true"` on submit button during bootstrap
- Error messages: `role="alert"` on inline error containers
- Bootstrap page title: `<title>Initialize AXIAM — AXIAM</title>`
