<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| application connection → audit_log table | Append-only invariant; only the GDPR path may UPDATE |
| DB at rest → provider secrets | SMTP password / API key must never be stored plaintext |
| pepper key → audit pseudonym | Re-identification resistance depends on pepper secrecy |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-secret-rest | Information Disclosure | email_config provider secrets (D-17) | mitigate | AES-256-GCM via encrypt_separate; ciphertext+nonce+key_version columns; dedicated AXIAM__EMAIL_ENCRYPTION_KEY (separate blast radius) |
| T-5-pseudonym | Information Disclosure | gdpr_pseudonym (D-02) | mitigate | Keyed HMAC-SHA256 with dedicated pepper; brute-force of candidate user_ids requires the pepper; 64-bit truncation retains correlation without re-identification |
| T-5-audit-update | Elevation of Privilege | audit_log UPDATE permission relaxation (D-04) | mitigate | Schema permission scoped to `gdpr_pseudonymizer` role (Option A); true enforcement is the single sanctioned repo method (Plan 05); FOR delete stays NONE |
| T-5-token-store | Spoofing | account_deletion / export_job tokens | mitigate | Store only token hashes (D-09/D-13); raw token returned once to caller |
| T-5-SC | Tampering | npm/cargo installs | mitigate | No new packages this phase — all deps workspace-existing (RESEARCH § Package Legitimacy Audit: N/A) |
</threat_model>
