Implement the AMQP mail consumer (D-14) — the single hardened delivery primitive — and wire it into the server: load the two new keys, run the email-secret backfill (D-17), and spawn the consumer. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AMQP message → EmailService | template context may carry user-controlled username |
| delivery failure → audit_log | must not introduce recipient PII |
| env → encryption keys | keys gate secret decryption and pseudonymization |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-template | Tampering | username in HTML body | mitigate | render_email/render_html HTML-escapes all context values (D-18) |
| T-5-addr-leak | Information Disclosure | email.delivery_failed metadata | mitigate | Audit keyed on user_id; to_address excluded from metadata (D-16) |
| T-5-mail-drop | Denial of Service | exhausted retries | mitigate | nack requeue:false → DLQ + delivery_failed audit; persistent publish |
| T-5-key-absent | Denial of Service | missing email key | accept | Consumer not spawned; warn logged; no secret decryption attempted |
| T-5-SC | Tampering | cargo installs | accept | No new packages — axiam-email/lapin/serde_json workspace deps |
</threat_model>
