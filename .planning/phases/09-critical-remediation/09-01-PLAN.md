<objective>
Close the cross-org IDOR defect (SEC-002): org-nested REST routes leak/modify data across organizations because they never compare the path `org_id` against the caller's JWT-derived `org_id`. Add the 4-line org-ownership guard (canonical pattern from `settings.rs:42-48`) to `organizations.rs`, `tenants.rs`, and `ca_certificates.rs`, restrict `organizations` create/list to system-admin, and add cross-org negative tests.

Purpose: A tenant admin in org A must not be able to read, modify, or delete org B's organization record, tenants, or CA certificates.
Output: Three hardened handler files + cross-org 403 test cases in three test files.
</objective>

Canonical org-scoping guard — VERIFIED from settings.rs:42-48:

    if org_id != user.org_id {
        return Err(AxiamApiError(
            axiam_core::error::AxiamError::AuthorizationDenied {
                reason: "cannot access a different organization".into(),
            },
        ));
    }

- `AuthenticatedUser.org_id` is type `Uuid`, sourced from the validated JWT `org_id` claim (extractors/auth.rs:79). All three handlers already extract `user: AuthenticatedUser`.
- `AxiamError::AuthorizationDenied` maps to HTTP 403 via the `ResponseError` impl on `AxiamApiError`.
- The guard has NO database round-trip; it precedes `repo.*` calls.
- super-admin detection pattern — VERIFIED auth.rs:650-664: iterate `role_repo.get_user_roles(user.tenant_id, user.user_id)`, set a flag when `role.name == "super-admin"`.
- tenants.rs already compares `tenant.organization_id != path.org_id` on get/update/delete (returns 404). The NEW requirement is to ALSO compare `path.org_id != user.org_id` (403) and to add the guard to create/list which currently have none.
- organizations create/list have NO path org_id (they operate on the collection), so they cannot use the org_id-mismatch guard — they need a system-admin role check instead.
