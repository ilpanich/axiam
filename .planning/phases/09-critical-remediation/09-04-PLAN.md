<interfaces>
Silent-refresh gap (VERIFIED api.ts:92-97): the refresh uses bare `axios.post("/api/v1/auth/refresh", {}, { withCredentials: true })`, bypassing the `api` instance request interceptor that attaches `X-CSRF-Token`. The refresh endpoint is CSRF-protected → 403 → clearAuth → /login.

Fix 1 (CSRF): use `await api.post("/api/v1/auth/refresh", {})` (the `api` instance already sets `withCredentials` and CSRF). Pre-set `_retry: true` on the refresh call so the response interceptor does not treat the refresh's own 401 as retriable (Pitfall 2 / Pitfall 5 — no loop).

Fix 2 (skip-list): replace `originalRequest.url?.includes("/auth/")` with an explicit list:
    const SKIP_REFRESH = ["/api/v1/auth/refresh", "/api/v1/auth/login", "/api/v1/auth/logout"];
    const isSkipRefresh = SKIP_REFRESH.some(u => originalRequest.url?.includes(u));
This keeps /refresh in the skip-list (no infinite loop) but allows a 401 from /auth/me to trigger refresh.

Boot-refresh gap (VERIFIED useAuthInit.ts:18-26 + fetchCurrentUser.ts): at boot, `fetchCurrentUser()` returns null on 401 and does not refresh because `isAuthenticated` is false. Add an explicit single boot refresh: if the first fetch returns null, `try { await api.post("/api/v1/auth/refresh", {}); user = await fetchCurrentUser(); } catch { /* genuinely unauth */ }`, then `user ? setUser(user) : clearAuth()`. Wrap in try/catch so a failed refresh calls clearAuth without re-entering the interceptor (Pitfall 5). The `setInitializing` no-op is pre-existing and out of scope.

CSRF contract test: extend frontend/e2e/auth-contract.spec.ts (created by 09-03) with a test asserting the refresh POST carries an `x-csrf-token` request header (route-intercept `**/auth/refresh`, read `route.request().headers()`).
