<objective>
Align all frontend list services to the backend list contract. The backend returns
`PaginatedResult<T>` = `{ items, total, offset, limit }` for tenant-scoped list
endpoints, but ~20 service methods are typed `Promise<T[]>` and return `r.data`
(the wrapper object) directly â€” so callers' `.filter`/`.map`/`.length` operate on an
object and crash or silently show nothing. Fix systemically with a defensive helper.
</objective>

<context>
Discovered during the Phase-13 manual smoke: DashboardPage crashed with
"(intermediate value).filter is not a function" because certificateService.list()
returned the PaginatedResult object. certificates/roles/groups list were already
fixed inline (commit ac0f8e7); this plan completes the sweep and refactors those
three to the shared helper.

TypeScript did not catch this because `api.get<T[]>(url)` CASTS the response to T[]
without validating the runtime shape. The fix types the response as the wrapper
union and unwraps.
</context>

  <files>frontend/src/services/_pagination.ts + the service files in frontmatter</files>
  <action>
  1. Create `frontend/src/services/_pagination.ts` exporting a defensive helper:
     ```ts
     /** Unwrap a list response that may be a bare array or a PaginatedResult ({ items }). */
     export function unwrapList<T>(data: T[] | { items?: T[] } | null | undefined): T[] {
       if (Array.isArray(data)) return data;
       return data?.items ?? [];
     }
     ```
     It is robust to BOTH shapes, so it is safe to apply even where an endpoint returns
     a bare array (it returns it unchanged).
  2. For EVERY service method typed `Promise<X[]>` that does `api.get<X[]>(url).then((r) => r.data)`,
     change to: `api.get<X[] | { items: X[] }>(url).then((r) => unwrapList(r.data))` and import unwrapList.
     Sweep all of: certificates (list â€” refactor the inline fix to the helper), roles (list, listPermissions,
     listUsers, listGroups, listByGroup), users (the `groupService.list`, `listMfaMethods`, `listMembers` â€”
