<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → `/api/v1/authz/check[/batch]` | Untrusted JSON body crosses; identity must come from JWT, not body |
| handler → AuthorizationEngine | Same decision path as gRPC; no divergent logic permitted (D-08) |
| handler → append-only audit log | Cross-subject queries must be recorded (legally significant) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | Elevation of Privilege | `subject_id` override in CheckAccessBody | mitigate | `RequirePermission::new("authz:check_as", user.tenant_id)` gate before any cross-subject decision; non-admin → 403 (Task 2, test b) |
| T-15-02 | Information Disclosure | `reason` field of deny response | mitigate | `reason` is the engine's generic deny string only; no resource-existence/structure hints added (Task 2) |
| T-15-03 | Spoofing | cross-tenant check via forged body | mitigate | `tenant_id` sourced exclusively from `user.tenant_id` (JWT); body has no tenant field (Task 2, acceptance) |
| T-15-04 | Repudiation | bypass of cross-subject audit | mitigate | audit entry (action `authz.check_as`) written on every override before returning (Task 2, test c) |
| T-15-05 | Denial of Service | authz-check used to enumerate/amplify | accept→mitigate | dedicated `authz_check_per_min` bucket (default 300) scoped to the two authz resources only — NOT a global ceiling raise (Task 1 + Task 3) |
| T-15-SC | Tampering | npm/cargo installs | n/a | No new runtime deps in this plan (RESEARCH Package Legitimacy Audit) |
</threat_model>
