<objective>
Implement the REST transport, token lifecycle, and local JWKS verification — the critical path of Phase 16. Covers the two-phase auth flow (login → verify_mfa), refresh/logout, the single-flight refresh guard, local EdDSA token verification, and the REST authz methods (check_access/can/batch_check), all enforcing §3 CSRF, §4 cookie-jar, §5 tenant context, §7 Sensitive<T>, and §9 single-flight from CONTRACT.md.

Purpose: This plan delivers success criteria #1 (typed LoginResult two-phase flow) and #2 (exactly-1-refresh single-flight) — the two highest-risk behaviors — and exposes the TokenManager + JwksVerifier that the gRPC and Actix plans consume.
Output: A usable REST client (`AxiamClient`) with proven login/MFA and single-flight refresh, plus the JWKS verifier reused downstream.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SDK client → AXIAM REST API | Untrusted network; TLS must be strict; tenant + CSRF headers cross here |
| cookie jar → token state | Raw access token extracted from jar must be wrapped in Sensitive<T> before any further handling |
| concurrent callers → refresh path | Thundering-herd refresh must be serialized to exactly one in-flight call |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-04 | Denial of Service | refresh path under concurrency | high | mitigate | Single-flight `tokio::sync::Mutex` guard with double-check; `single_flight_refresh_test` asserts exactly 1 call across 5 concurrent callers (§9) |
| T-16-05 | Spoofing / Tampering | TLS to REST API | high | mitigate | reqwest `rustls-tls`; strict verification always on; only escape hatch is `with_custom_ca` (PEM); NO insecure/skip surface (§6); CI grep gate in 16-06 |
| T-16-06 | Elevation of Privilege | cross-tenant access via wrong tenant header | high | mitigate | `tenant_slug`/`tenant_id` non-optional at construction; `X-Tenant-ID` injected on every request; server cross-validates against JWT claims (§5) |
| T-16-07 | Tampering (CSRF) | state-changing REST requests | high | mitigate | Capture `X-CSRF-Token`/`axiam_csrf` cookie and forward on POST/PUT/PATCH/DELETE (§3) |
| T-16-08 | Information Disclosure | access token in jar/refresh logs | high | mitigate | Token wrapped in `Sensitive<T>` immediately on jar extraction; no token in error messages or tracing (§7); no auto-retry that could log refresh token (§9.3) |
| T-16-09 | Spoofing | forged/altered access token verified locally | high | mitigate | Local EdDSA signature + exp verification against cached JWKS; reject `alg != EdDSA`; kid-rotation refetch (D-03/D-11) |
</threat_model>
