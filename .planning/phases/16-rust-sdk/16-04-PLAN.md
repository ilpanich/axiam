<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| RabbitMQ broker → SDK consumer | Messages are untrusted until HMAC-verified; an attacker who reaches the broker could inject spoofed/tampered messages |
| consumer → user handler | The handler is downstream trusted code; only verified messages may cross into it |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-10 | Tampering / Spoofing | AMQP message body | high | mitigate | Mandatory HMAC-SHA256 verify BEFORE handler invocation; constant-time `verify_slice`; byte-identical wire format to server; nack-without-requeue + security event on failure (§8/D-07) |
| T-16-11 | Information Disclosure | security-event / signing-key logging | high | mitigate | Signing key wrapped in `Sensitive<Vec<u8>>`; security-event log omits HMAC values and the key (§8.4); `Sensitive<T>` Debug redaction |
| T-16-12 | Denial of Service | poison/unverifiable messages requeued | medium | mitigate | Failures nack with `requeue: false` so bad messages do not loop the queue (§8.2g) |
| T-16-13 | Spoofing | missing-signature messages accepted | high | mitigate | Strict mode is the default — missing signature is rejected (nack-no-requeue), not log-and-accept (§8.3) |
</threat_model>
