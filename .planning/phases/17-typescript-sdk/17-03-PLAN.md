<objective>
Implement the Node persona's auth internals and gRPC transport (D-09/D-10/D-11/D-13/D-26): a
tough-cookie jar wired into the axios session so httpOnly cookies persist and the access/refresh
tokens can be read by name (the only token-source path, since login/refresh bodies carry no tokens);
a token manager wrapping token values in `Sensitive<T>`; local JWKS verification via `jose`'s
`createRemoteJWKSet` against `/oauth2/jwks` (EdDSA-only) for proactive refresh; and a `@grpc/grpc-js`
client with a reused channel, a synchronous auth/tenant interceptor, and a call-wrapper that handles
`UNAUTHENTICATED` by awaiting the shared single-flight refresh and retrying once. The Node session
extends the 17-02 `SharedSession` so one `login()` drives all transports (D-13).

Purpose: Deliver the Node persona's gRPC authz path (SC#2 Node half) and the proactive-refresh token
lifecycle, composing on the REST session so the single-flight guard is shared across REST + gRPC.
Output: `axiam-sdk/grpc` entry with a working checkAccess/batchCheck over gRPC, plus the Node
cookie-jar/token-manager/JWKS internals and their unit tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Node SDK -> gRPC AuthorizationService | Bearer token + tenant metadata cross here; must be current and correctly scoped. |
| server JWKS -> local verification | JWT signature/exp verified locally; algorithm confusion is the risk. |
| token cache -> logs/metadata | Token values must stay Sensitive; only expose() at the metadata-injection point. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-17-14 | Spoofing | JWT algorithm confusion (alg:none / HS-with-pubkey) | high | mitigate | jose jwtVerify with explicit `algorithms:['EdDSA']`; never trust token's own alg; test rejects non-EdDSA (D-11). |
| T-17-15 | Information Disclosure | token in gRPC metadata logs / cache | high | mitigate | Tokens wrapped in Sensitive<T>; expose() only at metadata.add; no token logging (D-26/§7). |
| T-17-16 | Denial of Service | UNAUTHENTICATED refresh storm across gRPC | high | mitigate | callWithRefresh shares the REST single-flight guard (D-13/§9); retry exactly once; no loop. |
| T-17-17 | Tampering | TLS bypass in gRPC channel | high | mitigate | Default TLS ChannelCredentials; customCa builds SSL creds from PEM only; no insecure credentials surface (§6; 17-06 lint gate enforces). |
| T-17-18 | Spoofing | stale token injected because start() cannot await | medium | mitigate | Interceptor reads only the sync cached token; refresh handled at callWithRefresh; syncFromJar after refresh keeps the cache current (Pitfall 3). |
</threat_model>
