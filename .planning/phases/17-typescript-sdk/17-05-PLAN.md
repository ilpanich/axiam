<objective>
Implement the Express and Fastify middleware (D-27 / §10) and the runnable examples that satisfy
SC#4. Both frameworks share one verification core that validates the incoming session against the
cached JWKS from 17-03 (no per-request server round-trip), injects the authenticated identity as
`req.axiamUser` (Express) / `request.axiamUser` (Fastify), and returns 401/403 on failure. Cookie
parsing is a tiny internal parser to avoid a `cookie-parser`/`@fastify/cookie` peer dependency
(dependency-minimalism, D-04 spirit; matches the Rust extractor's cookie-then-Bearer fallback).
The five examples (Express app, Fastify app, browser REST, Node gRPC, AMQP consumer) compile under
`strict` — the compile is the SC#4 gate.

Purpose: Complete the middleware contract (§10) and provide the runnable, strict-compiling examples
that prove SC#4 and demonstrate each persona/transport end to end.
Output: `src/middleware/{express,fastify}` + shared verify core, five example scripts, an examples
tsconfig with strict:true, and middleware unit tests (401 on missing/invalid, 200 + injected user on
valid).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| incoming HTTP request -> protected route | Untrusted session token crosses here; middleware verifies before the handler runs. |
| JWKS cache -> middleware decision | Stale/forged keys would allow forged sessions; explicit EdDSA + exp check mitigates. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-17-19 | Spoofing | forged/expired session at middleware | high | mitigate | Local jose verify (EdDSA + exp) before handler; 401 on failure; roles from verified scope claim only (D-27/§10). |
| T-17-20 | Elevation of Privilege | missing 403 on authz failure | medium | mitigate | AuthError->401, AuthzError->403 standardized bodies (§10). |
| T-17-21 | Information Disclosure | token echoed in middleware error body | medium | mitigate | Standardized JSON error bodies contain no token value; identity injection carries only userId/tenantId/roles. |
| T-17-22 | Tampering | stale JWKS cached beyond token TTL | low | accept | jose's exp enforcement bounds acceptance to the token's own lifetime; no extra caching layer added (§10). |
</threat_model>
