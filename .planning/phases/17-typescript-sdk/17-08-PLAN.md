<objective>
Close the remaining two confirmed Critical security gaps from 17-VERIFICATION.md:

- **CR-01:** `session.csrfToken` (rest/session.ts:25) has no writer in the Node path â€” every
  state-changing Node REST call sends an empty `X-CSRF-Token`, which the server's double-submit
  middleware rejects. Add a jar-read (mirroring `TokenManager.syncFromJar()`) that extracts the
  `axiam_csrf` cookie into `session.csrfToken`, invoked after login()/verifyMfa() succeed and
  after doRefresh(). Because auth.ts operates on the `SharedSession`-typed `client.session`
  (not `NodeSession` specifically), introduce a session-level `onAuthenticated` hook that
  NodeSession implements.
- **CR-04:** `NetworkError.cause` (core/errors.ts:38-39) carries the raw axios error verbatim;
  on error paths where the server already sent Set-Cookie (axiam_access/axiam_refresh), the raw
  cookie is reachable via `err.cause.response.headers['set-cookie']` through
  `console.log`/`JSON.stringify`/`util.inspect`. Add a `sanitizeAxiosError(err)` helper that
  strips Set-Cookie (and other sensitive response headers) before attaching as cause, at all four
  auth.ts call sites; update errorMapper.test.ts to assert redaction instead of verbatim
  preservation.

Both gaps modify `rest/auth.ts`, so they are sequenced tasks in ONE plan (CR-01 first, then
CR-04) to avoid edit conflicts. This plan is Wave 2 because it re-touches `rest/session.ts` and
`node/session.ts`, which plan 17-07 also modifies (per-session refresh guard); running after
17-07 keeps those two files conflict-free.

Purpose: make the Node persona's CSRF-protected endpoints actually work end-to-end (D-05) and
uphold the SDK's own no-raw-token-in-error-fields invariant (D-16).
Output: a Node CSRF sync via a session hook, a Set-Cookie-stripping error sanitizer wired into
all four auth call sites, one regression test per gap, and the errorMapper test updated to lock
in redaction.
</objective>
