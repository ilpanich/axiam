<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SDK error chain → consumer logs/telemetry | A `NetworkError` wrapping an `*http.Response` may carry `Set-Cookie` tokens that leak when the consumer logs the error |
| Sensitive value → any fmt/log/JSON surface | Token-carrying fields must never render their raw value through Go's formatting/serialization verbs |
| Generated stubs (build/CI tooling) → committed source tree | Committed `.pb.go` stubs must match the proto (drift = tampering risk / stale contract) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-18-01 | Information Disclosure | `NetworkError` wrapping `*http.Response` with `Set-Cookie`/`Authorization`/`Cookie` | high | mitigate | `sanitizeResponse` strips sensitive headers BEFORE wrapping (`newNetworkError`); regression test asserts raw token absent from `%v`/`%+v`/`%#v`/`json.Marshal`, with a non-vacuous control (D-04 / CR-04 carry-forward) |
| T-18-02 | Information Disclosure | `Sensitive` token fields via `fmt`/`json`/log | high | mitigate | `Sensitive` redacts across `String`/`Format`/`GoString`/`MarshalJSON`; raw value only via unexported `expose()`; test covers all four surfaces (D-08 / §7) |
| T-18-03 | Tampering | Committed gRPC stubs drifting from `proto/axiam/v1` | medium | mitigate | D-01 commits stubs + a CI `git diff --exit-code` drift-check job (added in Plan 06) so stubs always match the protos |
</threat_model>
