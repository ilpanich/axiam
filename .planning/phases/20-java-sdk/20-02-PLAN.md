<objective>
Resolve the single highest-risk correctness item of this phase FIRST: the §8 AMQP HMAC-SHA256 canonicalization. Implement `io.axiam.sdk.amqp.Hmac.verify(...)` using Jackson `ObjectNode` insertion-order preservation (NEVER alphabetical key sorting), prove it byte-for-byte against a real cross-language fixture whose signatures were produced by the canonical Rust signer (`crates/axiam-amqp/src/messages.rs`), and ship the `ErrDrop` poison-message sentinel the consumer (20-07) will use. This inherits Phase 19 (Python)'s empirically-proven ordering finding rather than re-deriving it — but re-proves it in Java against the same fixture so the AMQP consumer never builds on an unverified assumption.

Purpose: The §8 verify-before-handler contract and the entire AMQP subsystem depend on this canonicalization being byte-identical to the server. An alphabetizing canonicalizer would silently fail 100% of verifications, indistinguishable from a connectivity bug.
Output: A verified `Hmac` primitive, the `ErrDrop` sentinel, the reusable fixture, and a non-vacuous regression test.
</objective>

Read the `hmac_signature` text node; if absent/null, return false (§8.3 strict-mode default rejects missing signatures). Call `node.remove("hmac_signature")` — this mutates the `LinkedHashMap`-backed `ObjectNode` in place, PRESERVING the relative order of all remaining keys (the load-bearing property; do NOT create a sorted/TreeMap copy, do NOT enable any ObjectMapper key-sorting feature). Create `io.axiam.sdk.amqp.ErrDrop extends RuntimeException` with an `ErrDrop(String message)` constructor (poison-message sentinel; 20-07 maps it to nack-without-requeue).
