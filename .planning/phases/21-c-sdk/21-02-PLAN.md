<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| RabbitMQ broker → SDK consumer | Message bodies are attacker-influenceable in transit / at rest; only a valid server HMAC proves authenticity. |
| SDK consumer → application handler | The consumer-supplied handler must never execute on an unverified message. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-21-03 | Tampering | Amqp/Hmac.cs + AxiamAmqpConsumer.cs | high | mitigate | HMAC-SHA256 verify-before-handler with wire-order-preserving canonical JSON and `CryptographicOperations.FixedTimeEquals` (Task 1); handler structurally unreachable on failure; nack-without-requeue on any verification failure (Task 2). Proven against the shared Rust-signed fixture + fake-channel matrix tests. |
| T-21-04 | Information Disclosure | AMQP security log | medium | mitigate | Security-event log on HMAC failure carries only exchange/routing-key context — never the HMAC value or the message body (Task 2, PATTERNS.md lines 104-109 analog). |
| T-21-05 | Denial of Service | requeue loop on poison message | medium | mitigate | Drop-sentinel (`PoisonMessageException`) and HMAC/parse failures nack WITHOUT requeue, preventing an infinite redelivery loop; only genuinely-transient handler exceptions requeue (Task 2). |
</threat_model>
