<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Consuming app → AXIAM server (REST) | Requests carry bearer/cookie credentials + tenant context over TLS; a downgraded TLS config or dropped cookie jar breaks the security model. |
| Caller-supplied handler → SDK HttpClient | A caller may inject a handler missing cookies or weakening TLS. |
| SDK → authz decision | An authz result must reflect the server's live additive-only decision, never a stale client cache. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-21-10 | Tampering / Information Disclosure | AxiamHttpClientFactory (TLS) | high | mitigate | No `ServerCertificateCustomValidationCallback` bypass; the only escape hatch is an additive customCa chain-trust store; SDK re-applies strict TLS over any supplied handler (Task 1, D-09/§6/SC#4). Grep-gate previewed in Task 1, enforced in CI (21-07). |
| T-21-11 | Spoofing | Cookie jar dropped by override | high | mitigate | SDK always re-applies `UseCookies=true` + `CookieContainer` on both the owned and IHttpClientFactory paths (Task 1, §4) — an override can never silently unauthenticate the session. |
| T-21-12 | Elevation of Privilege | AuthzRestClient (stale authz cache) | medium | mitigate | No client-side authz caching / deny-override; every `CheckAccessAsync` calls the server fresh (Task 3, additive-only RBAC constraint). |
| T-21-13 | Information Disclosure | X-CSRF-Token / bearer injection | medium | mitigate | Access token read from the cookie jar and wrapped `Sensitive`; CSRF captured from the response header (§3) and echoed only on state-changing calls; errors redacted via ErrorMapper/NetworkError (Task 2). |
</threat_model>
