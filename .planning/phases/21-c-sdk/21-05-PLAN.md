<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Consuming app → AXIAM gRPC service | HTTP/2 authz calls carry bearer/tenant metadata over TLS; a bypassed TLS config or a per-transport refresh guard breaks the security/consistency model. |
| gRPC metadata → server cross-validation | The server rejects wire tenant_id/subject_id that disagree with the verified token claims. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-21-14 | Tampering / Information Disclosure | AxiamGrpcChannel (TLS) | high | mitigate | Channel reuses the REST factory's strict-TLS handler; no `ServerCertificateCustomValidationCallback` bypass; customCa additive-only (Task 1, §6/SC#4). Grep-gate covers `Grpc/`. |
| T-21-15 | Denial of Service | UNAUTHENTICATED refresh stampede over gRPC | high | mitigate | gRPC UNAUTHENTICATED retry uses the SAME `SemaphoreSlim(1,1)` `RefreshGuard` as REST (Task 1), so concurrent REST+gRPC 401/UNAUTHENTICATED still yield one refresh (D-10). Proven by the shared-guard retry test. |
| T-21-16 | Elevation of Privilege | gRPC wire tenant/subject spoof | medium | mitigate | Wire `tenant_id`/`subject_id` resolved from verified token claims, not a caller string; server cross-validates and rejects mismatch (Task 2). |
</threat_model>
