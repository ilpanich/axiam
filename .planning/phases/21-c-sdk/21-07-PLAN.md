<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SDK source tree → published package | A TLS-bypass introduced anywhere in package source would ship to every consumer; CI must gate it. |
| CI secrets → NuGet registry | The NuGet API key authorizes publishing under the reserved package IDs; it must never leak into logs and must be scoped to the publish job. |
| Example code → consumer copy-paste | Examples are widely copied; an insecure example pattern propagates. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-21-21 | Tampering / Information Disclosure | TLS-bypass anywhere in package source | high | mitigate | SC#4 gate enforced twice — a `TlsBypassGrepGateTests` xUnit test (Task 2) AND a CI grep-gate step (Task 3), both excluding only the additive customCa `CustomTrustStore` pattern; the job fails on any bare validation-disabling callback. |
| T-21-22 | Information Disclosure | NUGET_API_KEY in CI | high | mitigate | API key read only in the tag-triggered publish job via `${{ secrets.NUGET_API_KEY }}`, never echoed; push step guarded so an absent key degrades to a maintainer action rather than a broken/insecure fallback (Task 3, D-04). |
| T-21-23 | Tampering | insecure example pattern | medium | mitigate | Examples use only public entry points and the middleware's documented secure form; no TLS bypass, no manual JSON error concatenation; examples are inside the grep-gate/build scope (Tasks 1-2). |
| T-21-SC | Tampering | GitHub Actions pins | medium | mitigate | `actions/checkout` stays SHA-pinned; `actions/setup-dotnet` pinned by major-version tag with a documented maintainer follow-up to repin to SHA (no GitHub egress to source a SHA in-env — Python/Java precedent). |
</threat_model>
