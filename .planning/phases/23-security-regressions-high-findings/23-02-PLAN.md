<objective>
Close SEC-058: apply the tenant guard to the live REST grant path. `grant_to_role_with_scopes` (permission.rs:428-459) takes `_tenant_id` (underscore-prefixed, literally unused) and `RELATE`s across records with no tenant predicate on either branch. The SEC-007/CQ-B07 remediation landed only on the sibling `grant_to_role` (permission.rs:314-352); the REST-reachable scoped path was left unguarded, allowing a tenant-A `permissions:grant` holder to attach tenant-B permissions/scopes to a tenant-A role.

Purpose: The REST handler `POST /api/v1/roles/{role_id}/permissions` (permissions.rs:196-227) already passes `user.tenant_id` into `grant_to_role_with_scopes` — the bug is entirely the repo method ignoring it. Lift the proven `LET … IF array::len($x) = 0 { THROW }` in-query predicate from `grant_to_role` and extend it to also validate every scope id belongs to the caller's tenant.

Output: A tenant-guarded `grant_to_role_with_scopes` on both branches, and the `permission_grant_cross_tenant_rejected` test repointed at the vulnerable path with an added scope-ownership negative case.
</objective>
