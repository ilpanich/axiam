<objective>
Close SEC-059 and SEC-031: remove the `unwrap_or([0u8; 32])` all-zero webhook encryption-key fallback (main.rs:405-406) and actually encrypt webhook HMAC secrets on write. Today `encrypt_webhook_secret` (webhook.rs:250) is fully implemented and unit-tested but called from nowhere, so `create` stores the raw user secret verbatim (handlers/webhooks.rs:86-116) and a missing key silently degrades to all-zeros — a catastrophic key.

Purpose: Mirror PKI's proven fail-closed `Option<[u8;32]>` posture (main.rs:378-380). Per D-01 the server still boots without a key but refuses webhook register/deliver with an explicit error + `warn!`; per D-02 the secret is encrypted on create AND update and the update DTO gains secret rotation. Avoid the Actix `web::Data<T>` type-collision landmine: the email key is already registered as `web::Data<Option<[u8;32]>>` (main.rs:681), so the webhook key must NOT be registered as a second bare `Option<[u8;32]>` — route it through the uniquely-typed `WebhookDeliveryService`.

Output: fail-closed key handling, encrypt-on-write for create+update, a secret-rotation update DTO, and fail-closed + ciphertext-≠-plaintext negative tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Env var AXIAM__PKI__ENCRYPTION_KEY → webhook crypto | An absent/malformed key must disable the feature, never silently weaken it to a constant |
| Webhook create/update secret → DB storage | Plaintext HMAC secrets must be encrypted before crossing the DB boundary |
| Webhook row → API response | The stored secret must never be serialized back to any caller |
