<objective>
Close SEC-005 (XML Signature Wrapping): bind the verified XML signature to the assertion actually consumed by `handle_saml_response`, and enforce Destination + InResponseTo on the authenticated ACS path. Today `verify_signature` (saml.rs:378) returns only `Result<(), _>` — no reference binding — and the assertion is read from an independently-parsed `response.assertion` (saml.rs:436). An attacker can inject an unsigned sibling assertion that the parser picks up while a valid signature exists elsewhere in the document, and `verify_signature` still returns `Ok`.

Purpose: The correct fix is not hand-rolled reference matching — it requires `samael`'s XSW-safe `reduce_xml_to_signed` primitive, which does not exist at the pinned 0.0.19 and first ships in 0.0.20. This plan bumps `samael 0.0.19 → 0.0.21` (verified published + tarball-inspected in RESEARCH), migrates the one breaking call site, and parses the assertion from the reduced (verified-only) document. It also closes the authenticated-path Destination + InResponseTo gaps.

Scope boundary (LOCKED, per planner directive #2 and 23-RESEARCH Open Question 2 / Assumption A2/A4): the AUTHENTICATED ACS Destination check + InResponseTo requirement + XSW binding are IN SCOPE. The PUBLIC-path Destination fix that would require a `federation_login_state` schema migration (new `acs_url` column) is OUT OF SCOPE — deferred to the SEC-005 residual, recorded explicitly in Task 2's action, NOT silently dropped.

Output: samael bump + Cargo.lock + deny.toml re-verified; XSW-safe assertion extraction; authenticated-path Destination + InResponseTo; XSW/Destination/InResponseTo negative tests.
</objective>
