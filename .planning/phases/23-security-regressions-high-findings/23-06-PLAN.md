<objective>
Close SEC-044 / CQ-F27: the reset/resend flows 400 because the frontend omits backend-required `tenant_id` (and resend omits `email`). The backend DTOs are already correct (`RequestResetBody`/`ConfirmResetBody`/`VerifyEmailRequest`/`ResendVerificationRequest` all require `tenant_id`) — the gap is almost entirely frontend threading, plus a small backend addition to accept a tenant slug (so the URL-carried slug resolves) while preserving enumeration-safety.

Purpose: Per D-04, carry the tenant slug in the page URL (consistent with how login establishes tenant selection) and thread `tenant_id`/`email` into the three reset/verify service calls — `verifyEmail(tenantId, token)` (auth.ts:62-65) is already the correct-shape template. Per D-05, keep responses enumeration-safe: a slug-resolution failure must funnel into the SAME uniform `{sent:true}`/200 path as user-not-found, never a distinct status (Pitfall 4). Use query params for the three public routes (planner directive #3), matching `ResetPasswordPage`'s existing `?token=` convention.

Output: backend DTOs accept org/tenant slug with enumeration-safe resolution + `tenant_id` added to the email `template_context`; frontend threading across auth.ts/pages/router/LoginPage; and contract-test body assertions.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public reset/verify request → tenant resolution | Tenant context must come from the URL-carried slug or explicit id, resolved server-side |
| Reset/resend response → unauthenticated caller | The response must not reveal account existence, federation status, or slug validity |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-23-06-A | Information Disclosure (enumeration) | slug-resolution failure path (password_reset.rs:130-149) | high | mitigate | Route resolution failure into the uniform {sent:true}/200 path — indistinguishable from a valid account; enumeration-safety test (Task 1) |
| T-23-06-B | Denial of function (broken flow) | frontend omits tenant_id (auth.ts:39-74) | medium | mitigate | Thread tenant_id/email via URL-carried slug so requests match backend DTOs; contract body assertions (Task 3) |
| T-23-06-C | Information Disclosure | contract test blind to bodies | medium | mitigate | Assert request bodies (tenant_id/email) so a re-omission regression is caught |
| T-23-06-D | Spoofing (tenant confusion) | user-typed tenant / email-domain inference | medium | mitigate | Forbidden by D-04 — tenant only from URL slug / explicit id, never user-typed or inferred |

ASVS L1: V5 Input Validation (tenant resolved from URL slug / id, not free-typed), V7 Error Handling/Logging (enumeration-safe uniform responses, must not regress D-15). Block-on severity: high — the enumeration threat is mitigated with a proving test; no existing enumeration-safety is regressed.
