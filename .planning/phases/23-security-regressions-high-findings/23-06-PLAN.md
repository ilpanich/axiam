Forgot-password carries the slug in the page URL (D-04); confirm-reset/verify carry the raw tenant_id UUID (Open Question 2). Query params for the public routes (matching ResetPasswordPage's existing `?token=` convention).

Output: backend action_url construction in both handlers + optional-slug enumeration-safe tenant resolution on `RequestResetBody`; frontend threading across auth.ts/pages/router/LoginPage; a backend test proving the rendered link is substituted; and contract-test body assertions.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public reset/verify request → tenant resolution | Tenant context must come from the URL-carried slug or explicit id, resolved server-side |
| Reset/resend response → unauthenticated caller | The response must not reveal account existence, federation status, or slug validity |
| Server-built email link → user's inbox → SPA | The emailed action_url must be a fully-substituted, tenant-scoped link, not a template placeholder |

| T-23-06-B | Denial of function (broken flow) | frontend omits tenant_id/email (auth.ts:39-74) | medium | mitigate | Thread tenant context/email via URL-carried slug / raw tenant_id so requests match backend DTOs; contract body assertions (Task 3) |
