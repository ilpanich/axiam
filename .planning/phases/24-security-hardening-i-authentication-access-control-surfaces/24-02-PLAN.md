<objective>
Harden the public-path allowlist (SECHRD-11 / T19.25) so a non-canonical or wrong-segment request path cannot slip past `is_public_path` and reach a protected route as if it were public.

Purpose: `is_public_path` currently strips the trailing `*` and does a bare `path.starts_with(prefix)`, so a `/api/v1/auth/*` entry would also match `/api/v1/authz/...` (prefix confusion), and no normalization runs before the check, so `//` and `..` variants can dodge an exact-match entry. This is an Elevation-of-Privilege / Information-Disclosure access-control bypass.
Output: Segment-boundary-aware wildcard matching plus path normalization (collapse `//`, reject `..`) before the allowlist check, with negative unit tests asserting the matching-function property directly.
</objective>

  <action>
    In `is_public_path`, before the allowlist loop, normalize the path: collapse repeated slashes (`//` → `/`) and reject any path whose `/`-split segments contain a `..` (`path.split('/').any(|s| s == "..")` → return false, deny → falls through to the existing credential check). For `*`-suffixed entries, strip the `*` to get the prefix and require a segment boundary: match only if the remainder after the prefix is empty or begins with `/` (so `/api/v1/auth/` matches `/api/v1/auth/login` but not `/api/v1/authz/...`). Keep the exact-match arm for non-wildcard entries. Do NOT rewrite/canonicalize into a returned path — this function only returns a fail-closed bool; a normalization failure MUST return false (deny) so the request continues to the normal 401/403 credential check, never an implicit allow. Extend the inline `mod tests` with cases that call `is_public_path` directly (per the existing test style and Pitfall 6): assert `/api/v1/auth/*` does NOT match `/api/v1/authz/...` as a property of the function, plus `//` and `..` cases and a preserved-legitimate-wildcard case.
  </action>
