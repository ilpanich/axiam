<objective>
Fix the rate-limit client-IP keying bug (SECHRD-03 / SEC-048+060, decision D-01d): when there are not enough hops to trust, `XForwardedForKeyExtractor` currently falls back to `hops[0]` (the client-controlled leftmost `X-Forwarded-For` entry), letting an attacker rotate the header to get a fresh bucket per request and evade brute-force rate limiting.

Purpose: Close the rotating-XFF rate-limit-evasion bypass (Denial of Service / Spoofing) at the key-derivation layer. This plan is the keying fix only; the shared cross-replica store is plan 24-04.
Output: When `trusted_hops >= hops.len()`, ignore XFF entirely and fall through to `peer_addr()`; corrected nginx `trusted_hops` doc comment; a negative test proving rotating XFF yields the same bucket.
</objective>
