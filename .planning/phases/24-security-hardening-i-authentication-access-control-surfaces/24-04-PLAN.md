<objective>
Implement the multi-replica shared rate-limit store (SECHRD-03, decisions D-01a shared SurrealDB store / D-01b fail-open with per-replica in-memory fallback). Under HPA, per-replica in-memory buckets multiply the effective limit by the replica count; this closes that gap without new infra (reuse SurrealDB, no Redis).

Purpose: A brute-force attacker spread across replicas currently gets N× the intended attempts. A shared counter closes that. But a counter-store outage must never hard-block all auth traffic, so the design fails OPEN to the existing in-memory governor (the ONE deliberate fail-open exception in this phase).
Output: A `rate_limit_bucket` table, a windowed compare-and-set counter repository, and a NEW async Actix pre-check middleware that runs BEFORE the existing `Governor`/`GovernorLayer` and falls back to it on any DB error — with a cross-instance shared-store test.

CRITICAL (RESEARCH Pitfall 1): `governor::StateStore::measure_and_replace` is a synchronous trait method. Do NOT implement `StateStore` for a SurrealDB adapter and do NOT `block_on` inside it. Build a separate async layer that runs the SurrealDB CAS before delegating to the untouched in-memory governor.
</objective>
