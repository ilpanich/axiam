<objective>
Make the GDPR erasure audit-write durable (SECHRD-12 / T19.27, decision D-02): when the erasure audit DB-write fails, dead-letter the record to BOTH an append-only local file (on a mounted volume) AND a structured audit log event, so a legally-significant erasure event survives a transient DB failure even if one sink is absent.

Purpose: A transient SurrealDB failure during the erasure audit write currently silently loses a legally-significant record (Repudiation). D-02 requires two durable sinks.
Output: A dead-letter path on the audit-write-failure branch in `cleanup.rs`, an injectable audit-write seam so the failure is testable, and an integration test asserting both sinks fire.

RECORDED DECISION (resolves RESEARCH Assumption A4): "structured audit syslog" is satisfied by a structured `tracing` JSON audit event captured by the container log driver — NOT a literal `syslog(3)` socket. Rationale: the distroless deployment (per 06-01 decisions) has no local syslogd, and a real syslog sink would require a new crate; a structured tracing event is SIEM-ingestible and adds no dependency. The append-only file is the second, DB-independent durable sink.
</objective>
