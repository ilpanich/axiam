<objective>
Bring the gRPC rate limiter to parity with the fixed REST limiter (SECHRD-03, decision D-01c; resolves RESEARCH Open Question 1 = fix it): swap the gRPC limiter onto the shared SurrealDB store (D-01a/b) AND replace `tower_governor::SmartIpKeyExtractor` — which unconditionally trusts the leftmost `X-Forwarded-For` hop — with a custom `trusted_hops`-aware key extractor mirroring the REST fix.

Purpose: The gRPC authz limiter is currently keyed by the naive `SmartIpKeyExtractor` (no `trusted_hops` concept) and uses per-replica in-memory buckets — the same IP-spoofing and multi-replica gaps the REST plans close. The phase goal ("resists IP-spoofing") requires gRPC parity.
Output: A custom gRPC key extractor keyed off the verified peer address when hops are untrustworthy, a shared-store pre-check reusing the plan-24-04 repository with fail-open fallback, and a gRPC negative test.

HARD CONSTRAINT (D-01c coordination note): Do NOT touch the gRPC throughput/quota math. `Quota::per_second(authz_per_sec).burst_size(authz_per_sec * 2)` (~rate_limit.rs:40-47) is already correct in live code and is owned by CORR-01/Phase 26 — leave it byte-for-byte. This plan is store + key-extractor ONLY.
</objective>
