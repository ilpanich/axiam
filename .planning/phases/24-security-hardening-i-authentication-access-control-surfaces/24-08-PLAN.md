<objective>
Close the bootstrap initialized-check TOCTOU and make the first-run gate mandatory (SECHRD-04 / SEC-049, decisions D-03a, D-03b, D-03c): make first-super-admin creation atomic via a uniqueness-invariant CREATE inside the admin-creation transaction (D-03c — two concurrent first-run requests create at most one super-admin), and refuse bootstrap unless a gate (env var OR a server-minted one-time setup token) is satisfied.

Purpose: Today `bootstrap.rs` does a SELECT-then-branch check (list roles → find super-admin → list users → `total > 0`) BEFORE creating the admin — a TOCTOU window where two concurrent first-run requests can both create a super-admin (Elevation of Privilege). And the env gate is only enforced `if let Ok(expected) = env::var(...)`, so an UNSET var currently allows arbitrary bootstrap.
Output: New `bootstrap_lock` / `bootstrap_setup_token` / `bootstrap_setup_token_consumed` tables, a server-minted setup token logged once at first boot, a mandatory gate that fails closed, an atomic uniqueness-invariant bootstrap transaction (also seeding the admin's initial password into history), and concurrency + gate-unset negative tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| unauthenticated caller → public bootstrap endpoint | Anyone can hit bootstrap before an admin exists |
| concurrent requests → SurrealDB | Two first-run requests may race to create the first super-admin |

## STRIDE Threat Register (ASVS V4 Access Control, V11 Business Logic)

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-24-81 | Elevation of Privilege | TOCTOU in bootstrap.rs (SELECT-then-branch) | critical | mitigate | Uniqueness-invariant `bootstrap_lock` CREATE inside the admin transaction; concurrency test (Task 3) |
| T-24-82 | Elevation of Privilege | unconditional bootstrap when env gate unset | high | mitigate | Mandatory fail-closed gate: env var OR valid one-time setup token; gate-unset refusal test (Task 2) |
| T-24-83 | Spoofing (replay) | setup-token reuse | medium | mitigate | Consumption-by-existence: `bootstrap_setup_token_consumed` CREATE in the same transaction (Task 3) |
| T-24-84 | Information Disclosure | plaintext setup token in logs | low | accept (deliberate one-time exception, D-03b) | Single first-boot log line only; only the hash is persisted (Task 1) |
</threat_model>
