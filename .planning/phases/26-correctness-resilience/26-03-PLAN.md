<objective>
Fix CORR-03 (part 1 of 2): the webhook delivery path today is a detached `tokio::spawn` with an in-process sleep retry loop and ZERO call sites — it dies on restart and never runs. Build the durable-delivery primitives: split `WebhookDeliveryService::deliver()` into a publish-only `emit()` and a single-attempt `deliver_once()` (D-06), upgrade the signature to the Stripe-style signed-timestamp scheme (D-10), and add a correctly-wired webhook AMQP topology (primary/retry/DLQ) + message DTO + publisher (D-07). The consumer that drives this and the main.rs wiring land in plan 26-07 (Wave 2).

Purpose: Establish the signer + transport so a durable, retrying, auditable delivery path can be assembled without an in-process sleep tying up a consumer slot.
Output: The split delivery service, the new signature, and the AMQP webhook topology/publisher — all unit-testable without a live broker.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AXIAM -> external webhook endpoint | Signed payload leaves the trust boundary; signature integrity + SSRF safety must hold |
| stored webhook secret -> in-memory HMAC | AES-256-GCM-encrypted secret (SEC-031) is decrypted only in memory for signing |
| producer -> AMQP broker | Delivery messages must be durable and correctly routed (no silent drop) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-26-03-01 | Tampering/Spoofing | signature scheme change (D-10) | high | mitigate | HMAC-SHA256 over `t.body` with the per-webhook secret; timestamp bound into the signed payload (Stripe scheme) prevents body-only forgery and enables receiver-side replay windows; keeps the existing MAC primitive (no hand-rolled crypto) |
| T-26-03-02 | Information Disclosure | secret decrypt for signing | high | mitigate | Reuse SEC-031 decrypt-in-memory; secret never serialized in responses or logged; deliver_once holds it only for the single attempt |
| T-26-03-03 | Tampering (SSRF) | deliver_once fetch | high | mitigate | deliver_once routes through the existing `ssrf::guarded_fetch` (SECHRD-02 IP-pinning) unchanged — the guard is preserved when delivery moves to the queue-driven path |
| T-26-03-04 | DoS / silent data loss | retry/DLQ topology wiring | high | mitigate | Use the default-exchange + `x-dead-letter-routing-key` DLX form so retry/terminal messages are actually routed (Pitfall 4), not silently dropped by an undeclared-exchange DLX |
| T-26-03-SC | Tampering | cargo installs | low | accept | No new packages this phase (RESEARCH Package Legitimacy Audit: none) |
</threat_model>
