The NEW webhook topology deliberately avoids this (default-exchange + `x-dead-letter-routing-key` form). Fixing the pre-existing queues is NOT in this phase — file a follow-up tracking issue to verify/repair them against a live RabbitMQ instance in a future phase.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AMQP broker -> webhook consumer | Queued delivery messages (incl. redeliveries and poison messages) cross into the consumer; must not loop forever or bypass the SSRF/signature controls |
| consumer -> external webhook endpoint | Each attempt performs an outbound signed POST; SSRF + signature integrity must hold on every attempt |
| consumer -> audit trail | Delivery outcomes must be durably recorded (append-only) for every attempt |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-26-07-01 | Tampering (SSRF) | deliver_once from the queue | high | mitigate | The queue-driven path calls the SAME ssrf::guarded_fetch (SECHRD-02 IP-pin) via deliver_once — the guard is preserved when driven from AMQP, not bypassed |
| T-26-07-02 | DoS / poison message | consumer loop | high | mitigate | Malformed messages nack requeue:false (no infinite requeue); retries are bounded by max_attempts and dead-letter to WEBHOOK_DLQ; delays use the retry-queue TTL so no consumer slot is held (D-07) |
| T-26-07-03 | Information Disclosure | webhook secret in consumer | high | mitigate | Secret is decrypted in-memory (SEC-031) only inside deliver_once for the single attempt; never logged or persisted in the message/audit metadata |
| T-26-07-04 | Repudiation | missing delivery audit | medium | mitigate | Per-attempt + terminal audit records (D-09) written for success, retry, and exhaustion via the append-only audit repository |
| T-26-07-05 | Spoofing/Tampering (replay) | signed delivery | high | mitigate | Stripe-style t=,v1= signature (from 26-03) binds the timestamp into the HMAC; receiver-side replay-window enforcement is deferred (CONTEXT Deferred Ideas), sender emits the signed timestamp now |
| T-26-07-SC | Tampering | cargo installs | low | accept | No new packages this phase |
</threat_model>
