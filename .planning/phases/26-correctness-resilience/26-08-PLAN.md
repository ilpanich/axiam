<objective>
Fix CORR-05 frontend half (CORR-05b): an MFA-mandated login currently does `navigate("/profile/mfa", { state: { setup_token } })`, but `/profile/mfa` is inside `AppLayout` (auth-guarded) and only calls the self-service enroll/confirm endpoints — the router-state `setup_token` is lost on the guard redirect and refresh/bookmark, dead-ending the user on the login screen (D-16). Add a dedicated public `/auth/mfa-setup` route (sibling to `/auth/reset-password`, outside the auth guard) that carries `setup_token` as a URL query param, drives the `/mfa/setup/enroll` + `/mfa/setup/confirm` endpoints, reuses the extracted TOTP QR/secret/code UI, and on success hydrates tenant context via `fetchCurrentUser()` (relying on the 26-05 `/auth/me` slugs) and proceeds to the app. Also add the e2e proof that tenant context restores after a hard reload (backend from 26-05) and that MFA-setup has no dead end.

Purpose: Give MFA-mandated users a bookmark/refresh-safe path into the app, and confirm tenant-restore-after-reload works end-to-end.
Output: The new route + shared panel + service methods + router/LoginPage wiring + e2e coverage.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| unauthenticated setup-token carrier -> /auth/mfa-setup | The setup_token is a login-issued bearer for completing MFA enrollment; it must not be leaked/logged and the route must be reachable without a full session |
| /auth/me slugs -> Topbar restore | Restored tenant context derives from server-emitted slugs (26-05), scoped to the authenticated user |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-26-08-01 | Information Disclosure | setup_token in URL/logs | high | mitigate | setup_token is carried as a query param only for the enroll/confirm calls; it is not written to any console/log statement, and the page strips it from the URL bar after use (history.replaceState) mirroring ResetPasswordPage — no persistent secret leak |
| T-26-08-02 | Spoofing/Elevation | public /auth/mfa-setup route outside the auth guard | high | mitigate | The route only accepts a server-issued setup_token and calls the setup/* endpoints which validate it server-side; enroll/confirm failures (invalid/expired/used token) render the invalid-link state and force a fresh login — no bypass of authentication |
| T-26-08-03 | DoS (double-enroll) | StrictMode auto-enroll | medium | mitigate | useRef once-guard fires the enroll call exactly once per mount so a single-use setup_token is not consumed twice |
| T-26-08-04 | Repudiation/UX dead-end | mfa_setup_required stranded on login | medium | mitigate | Every reachable state has one clear action (retry via fresh login, or continue the form); the query-param (not router-state) carrier is bookmark/refresh-safe (D-16) |
| T-26-08-SC | Tampering | npm installs | low | accept | No new packages this phase (qrcode.react/react-router-dom already present) |
</threat_model>
