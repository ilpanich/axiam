# Phase 26 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note (read first):** Phase 26 is a **correctness phase**, not a net-new-design
phase. Only CORR-05 and CORR-06 touch the frontend; both are **behavioral correctness
fixes to existing flows** against AXIAM's existing dark neon-glass design system
(React 19 + TypeScript + Tailwind + Radix primitives under `frontend/`). The Design
System / Spacing / Typography / Color sections below document what already exists and
is **LOCKED** — nothing in this phase changes tokens, palette, or component visual
language. The load-bearing content of this contract is the **Interaction & State
Contracts** section, which specifies the exact states the CORR-05/06 fixes must
satisfy in the five touched surfaces: Topbar, the new MFA-setup landing, VerifyEmail,
Dashboard, and the org-settings form.

## Interaction & State Contracts (CORR-05 / CORR-06)

This section is the primary deliverable for this phase. It is normative for the
executor and the basis for `gsd-ui-checker`/`gsd-ui-auditor` verification.

### 1. Topbar tenant restore after hard reload (CORR-05, D-14/D-15)

**Component:** `frontend/src/components/layout/Topbar.tsx` (unchanged — already
reads `tenantSlug`/`orgSlug` from `useAuthStore`)
**Upstream fix locations:** `crates/axiam-api-rest/src/handlers/auth.rs`
(`LoginUserInfo`/`MeResponse` must add `tenant_slug: Option<String>` and
`org_slug: Option<String>`, populated in the `me` handler); frontend
`fetchCurrentUser.ts` and `useAuthInit.ts` already read these fields defensively —
**no frontend change required** once the backend emits them (verified by reading
both files: `fetchCurrentUser` already maps `res.data.tenant_slug ?? res.data.user?.tenant_slug`
and `useAuthInit` already calls `setTenantContext` when both slugs are present).

| State | Trigger | Visual contract |
|-------|---------|------------------|
| Restored | Hard reload; `/auth/me` returns `tenant_slug`+`org_slug` | Tenant selector button shows `{orgSlug} / {tenantSlug}` exactly as after a fresh login — **no flash of "Select tenant"** before the restored value renders (the restore must complete before `isInitializing` flips false and the authenticated shell renders) |
| Degraded (no slugs) | `/auth/me` succeeds but omits slugs (legacy session, org/tenant deleted, etc.) | Tenant selector shows the existing fallback **"Select tenant"** label — unchanged from current behavior. This is NOT an error: no alert, no toast, no console warning surfaced to the user |
| Degraded (me fails entirely) | `/auth/me` and the one-shot refresh both fail | Existing `clearAuth()` → redirect to `/login` path, unchanged |

**Non-negotiable:** the tenant-restore fix is additive and must never cause the app
shell to render, then flicker, then re-render with a different tenant label. The
Topbar's tenant value must be correct on first paint of the authenticated shell.

### 2. MFA-setup landing (CORR-05, D-16) — new route, no dead end

**Problem being fixed:** `LoginPage.handleCredentialsSubmit` currently does
`navigate("/profile/mfa", { state: { setup_token: data.setup_token } })` when
`mfa_setup_required` is true. `/profile/mfa` renders inside `AppLayout`, which
redirects unauthenticated users straight to `/login` (`AppLayout.tsx` line 21) —
the router-state payload carrying `setup_token` is lost and the user lands back on
the login screen with no path forward. `MfaManagementPage` also only calls the
**self-service** `/api/v1/auth/mfa/enroll` + `/api/v1/auth/mfa/confirm` endpoints,
which require an authenticated session — not the `/api/v1/auth/mfa/setup/enroll` +
`/api/v1/auth/mfa/setup/confirm` endpoints that accept a bare `setup_token`.

**Required fix:** a new **public** route (sibling to `/auth/reset-password` and
`/auth/verify-email`, NOT nested under `AppLayout`/`ProtectedRoute`) that carries
the `setup_token` as a URL query parameter (bookmark/refresh-safe — mirrors the
`ResetPasswordPage` `?token=&tenant_id=` pattern, per D-16), not as ephemeral router
`state` (router `state` is lost on refresh/bookmark — this is the actual root cause
of the dead end and must not be repeated).

- **Route:** `/auth/mfa-setup?setup_token=<token>`
- **Backend calls:** `POST /api/v1/auth/mfa/setup/enroll` (body `{ setup_token }`) on
  mount → `POST /api/v1/auth/mfa/setup/confirm` (body `{ setup_token, totp_code }`)
  on form submit
- **`LoginPage` change:** replace the `/profile/mfa` + router-state navigate with
  `navigate(\`/auth/mfa-setup?setup_token=${encodeURIComponent(data.setup_token)}\`)`

**Visual contract — reuse `PublicLayout` + the existing `TotpSetupDialog` QR/secret
presentation inlined as a page body (not a modal, since there is no authenticated
shell to host a dialog over):**

| State | Condition | Visual contract |
|-------|-----------|------------------|
| `no-token` | `setup_token` query param missing | Same pattern as `VerifyEmailPage`'s `no-token` state: muted icon (`MailOpen`-equivalent — use `AlertCircle` or `ShieldAlert`), heading **"Invalid setup link"**, body copy from Copywriting Contract, single outlined button **"Back to Login"** → `/login` |
| `loading` (enrolling) | Token present, `POST .../mfa/setup/enroll` in flight (auto-fired on mount) | Centered `Loader2` spinner (same 56px/`h-14 w-14` circular badge + `animate-spin` pattern as `VerifyEmailPage`'s loading state) with caption **"Preparing your authenticator setup…"** |
| `enroll-error` | Enroll call fails (expired/invalid/already-used token → 401) | Same visual pattern as the `no-token` state (destructive icon, "Invalid setup link" heading, same body copy) — an expired token and a missing token are indistinguishable to the user and must look identical |
| `ready` (QR + form) | Enroll succeeded | QR code (`QRCodeSVG`, same 160–176px sizing, white bg / dark fg as `TotpSetupDialog`) + manual secret entry with copy button (identical markup/behavior to `TotpSetupDialog`'s manual-entry block) + 6-digit code input (same `tracking-[0.5em] font-mono text-center text-2xl` styling as `LoginPage`'s MFA step / `TotpSetupDialog`'s verify input) + primary button **"Confirm & Continue"**, disabled until 6 digits entered |
| `confirming` | Confirm submitted, request in flight | Button shows `Loader2` spinner + **"Confirming…"** (same disabled+spinner pattern as every other form submit in the app — `ResetPasswordPage`, `LoginPage`) |
| `confirm-error` | Confirm call fails (wrong code, expired token mid-flow) | Inline `role="alert"` destructive banner above the code input, copy per Copywriting Contract — **the form stays on screen** (code field cleared or left for correction — do not bounce the user back to `no-token`/error screen on a wrong-code entry, only on token-level failures) |
| success | Confirm succeeds | No separate success screen (see Copywriting Contract) — hydrate auth store and `navigate("/dashboard")` immediately, identical to `LoginPage.handleMfaSubmit`'s success path |

**Non-negotiable:** at no point does a `mfa_setup_required` login response leave the
user on the login screen with no actionable next step (the defect this fix closes).
Every reachable state above has exactly one clear action (retry via a fresh login,
or continue the setup form).

### 3. VerifyEmailPage StrictMode once-guard (CORR-06, D-17)

**Component:** `frontend/src/pages/auth/VerifyEmailPage.tsx`
**Problem:** the existing `useEffect` uses a `cancelled` closure flag, which
survives React 18/19 StrictMode's mount→unmount→remount cycle in development
(cleanup runs, `cancelled` is set, but the **second** mount runs `doVerify()` again
with a fresh `cancelled=false`) — so the single-use verification token can be
consumed twice, and the first (aborted) response is discarded while the second
real network call may legitimately 401 with "already verified/used", surfacing a
false **error** state for what was actually a successful verification.

**Required fix (per useAuthInit.ts's existing proven pattern in this same
codebase — reuse it, do not invent a new pattern):** add a `useRef(false)`
once-guard exactly like `useAuthInit.ts`'s `initialized` ref — set it inside the
effect body before the async call fires, so the second StrictMode mount is a no-op
and the network call fires exactly once per real mount.

| State | Visual contract (unchanged from current implementation) |
|-------|-----------------------------------------------------------|
| `no-token` | Existing markup/copy — unchanged |
| `loading` | Existing markup/copy — unchanged |
| `success` | Existing markup/copy — unchanged |
| `error` | Existing markup/copy — unchanged, but must now be **unreachable** for the StrictMode-double-fire case; only a genuinely invalid/expired/reused token reaches this state |

**Non-negotiable:** no visual states change. This is a pure idempotency fix — the
UI contract is "the existing five states, but `error` is never falsely reached due
to double invocation." The `useRef` guard must not prevent the effect from firing
on genuinely distinct token/tenant param changes (e.g. two different verification
links loaded in the same SPA session are out of scope — full page nav is the normal
flow for email links).

### 4. Dashboard query-key collision (CORR-06, D-18)

**Component:** `frontend/src/pages/DashboardPage.tsx` (query key only —
`["users", 1, ""]`) vs. `frontend/src/pages/users/UsersPage.tsx`
(`["users", page, search]` with `userService.list(page, 20, search)`).
**Problem:** Dashboard's stat-card query calls `userService.list(1, 1, "")`
(limit 1, for the total count only) under the key `["users", 1, ""]`. When
`UsersPage` is on page 1 with no search filter, its key is also `["users", 1, ""]`
— TanStack Query treats them as the **same cache entry**, so whichever page
fetches last overwrites the cached response the other reads, producing a
1-item-limited result on the Users list or a stale/wrong total on the Dashboard
stat card depending on navigation order.

**Required fix:** give the Dashboard's total-count query a **key shape that can
never collide** with `["users", page, search]`, e.g. `["users", "dashboard-count"]`
or `["dashboard", "users-count"]` (exact string is Claude's discretion per D-18 —
the contract requirement is structural, not the literal string).

**Visual contract — no visible change to either page.** This is purely a
data-correctness fix; the constraint for the executor/checker is:

- The Users stat card on `DashboardPage` (`value={usersQ.data?.total ?? null}`)
  must continue to show the loading skeleton (`animate-pulse` gray bar) while
  loading and the correct total once loaded — same as today.
- Navigating Dashboard → Users → Dashboard (or vice versa) must never show a
  visibly wrong count (e.g. "1") on either page's Users number due to cache
  cross-contamination. This is the acceptance-visible symptom to check for.

### 5. Org-settings form: init-guard + dirtiness + navigate-away warning (CORR-06, D-19)

**Component:** `frontend/src/pages/organizations/OrganizationDetailPage.tsx`,
`SettingsTab` function (query key `["org-settings", orgId]`) — **this is the
"org-settings form" referenced by CORR-06**, confirmed by reading the file: it is
literally the only form in the codebase backed by an `org-settings` query key.
**Problem:** `useEffect(() => { if (settings) setForm(flattenOrgSettings(settings)); }, [settings])`
unconditionally re-seeds local form state from **every** query-data change,
including a background refetch triggered by TanStack Query's default
`refetchOnWindowFocus`. A user mid-edit who alt-tabs away and back loses all
unsaved field changes with no warning — the form silently reverts to server state.

**Required fix — two independent guards:**

1. **Init-once guard:** seed `form` from `settings` only on the *first* successful
   load per `orgId` (e.g. a `useRef`/`initializedOrgId` guard, or deriving initial
   state lazily and never re-seeding from a later `settings` change while the form
   is mounted for the same `orgId`). Switching to a **different** `orgId` (new
   `SettingsTab` mount) is expected to re-seed — this guard is per-mount/per-org,
   not permanent.
2. **Dirty tracking:** track whether `form` differs from the last-loaded server
   snapshot. Adds a **warn-on-navigate-away** guard beyond the D-19 minimum (per
   CONTEXT.md, explicitly in scope):
   - **In-app navigation** (router-level, e.g. clicking a different tab/sidebar
     link while dirty): show the destructive confirm dialog from the Copywriting
     Contract (title **"Discard unsaved changes?"**) — reuse the existing
     `ConfirmDialog` component (already used elsewhere in this codebase, e.g.
     `MfaManagementPage`'s delete-method dialog) for visual consistency, do not
     build a new dialog primitive.
   - **Browser-level navigation** (refresh/close tab): a `beforeunload` handler
     that sets `event.returnValue` when dirty — no custom UI possible here, browser
     chrome only (see Copywriting Contract note).

| State | Visual contract |
|-------|------------------|
| Clean (no unsaved edits) | Unchanged from today — no dirty indicator, no navigation guard active |
| Dirty (edits pending, not saved) | Add a small **dirty indicator** next to the existing "Save"/form action area — reuse the amber/warning token (`#f59e0b`, the same color already used for the cert-expiry warning banner elsewhere in the app) as a small dot or "Unsaved changes" label; do not introduce a new color for this |
| Dirty + attempted in-app navigation away | `ConfirmDialog`-styled modal per Copywriting Contract; "Discard changes" (destructive variant) proceeds with navigation and discards edits; "Keep editing" (outline variant) cancels navigation, dialog closes, form state untouched |
| Dirty + browser refresh/close attempt | Native `beforeunload` prompt (browser-controlled copy) |
| Background refetch while dirty (e.g. window refocus) | **No visible change** — the in-progress edited fields must NOT revert; only fields the user has not touched may reflect fresher server data (or, more simply/robustly per the init-once guard, the form is fully frozen from server updates for the remainder of the edit session — either approach satisfies this contract; freezing is preferred as the simpler, more predictable option) |
| Save succeeds | Existing success feedback (`setSaveSuccess(true)`, auto-clears after ~3s) — unchanged. Dirty state resets to clean and the navigate-away guard deactivates |

**Non-negotiable:** the fix must not remove the ability to see fresh server data
when the form is genuinely clean (e.g. another admin changed a setting and the user
hasn't started editing yet) — only edits-in-progress are protected.
