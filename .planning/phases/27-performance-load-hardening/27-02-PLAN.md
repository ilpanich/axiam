<objective>
Apply the uniform hand-rolled JWKS single-flight guard (D-08) to the Rust and Python
SDK verifiers so a burst of concurrent invalid-`kid` (cache-miss) verifications
collapses to exactly one network fetch (D-09). This is a coalescing WRAPPER only — it
must not touch the underlying JWT/JWK verification (`jsonwebtoken` / `PyJWT`).

Purpose: Close PERF-03 for 2 of the 7 SDKs. Protects the JWKS endpoint from
self-inflicted load (a rotating/forged-`kid` attack, or many clients hitting one app
instance right after a real key rotation).
Output: fetch-guard + double-checked cache re-read in each verifier, each proven by a
counting-mock test asserting exactly one fetch under a concurrent burst.
</objective>

- Python: `_refetch_lock` (existing `threading.Lock`) scope widened to cover the entire fetch
- New test file: `sdks/rust/tests/jwks_single_flight_test.rs`
- Extended test: `sdks/python/tests/test_jwks.py` (new burst case)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| resource-server (embedding the SDK) → AXIAM JWKS endpoint | Outbound key fetch on cache miss; an invalid-`kid` burst can turn this into a self-inflicted fetch storm |
| untrusted bearer token → SDK verifier | Attacker-controlled `kid`/token drives cache-miss fetches |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-27-20 | Denial of Service | JWKS endpoint under invalid-`kid` burst | medium | mitigate | Single-flight collapses N concurrent cache-misses to 1 fetch (Tasks 1-2), on top of the existing forced-refetch cooldown |
| T-27-21 | Spoofing / Tampering | single-flight caching a JWKS across a key rotation such that a revoked/rotated key is honored | high | mitigate | The guard wraps ONLY the fetch; freshness/TTL + forced-refetch-on-unknown-kid + the cryptographic verify are unchanged — acceptance criterion asserts the verify path is untouched, so no revoked key is honored beyond existing semantics |
| T-27-22 | Tampering | accidental modification of the crypto verify while adding the wrapper | high | mitigate | Acceptance criteria explicitly require the `jsonwebtoken`/`PyJWT` decode/verify call to be unchanged; full SDK suites must stay green |
</threat_model>
