<objective>
Parallelize `BatchCheckAccess` (gRPC + REST) with bounded, order-preserving concurrency
via `futures::stream::buffer_unordered`, bounded by a new `AXIAM__AUTHZ__BATCH_MAX_CONCURRENCY`
knob (default 16, deliberately under the ~30-connection SurrealDB budget) (D-06/D-07).
Create the new `AuthzConfig` section (it does NOT exist yet) mirroring `GrpcConfig`.

Purpose: Close PERF-02. Today both handlers evaluate items in a sequential
`for check_req in req.requests` loop; an N-item batch pays ~N×RTT. Concurrent evaluation
overlaps the I/O-bound DB round-trips toward ~(N/concurrency)×RTT. Correctness
(order + per-item equivalence) is the security gate — concurrency must introduce no
authorization bypass.
Output: `AuthzConfig` + `futures` direct dep + concurrent batch in both handlers + a
correctness test per transport.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → BatchCheckAccess (gRPC/REST) | Untrusted batch of access requests; each item carries a tenant_id/subject_id that must match the caller's verified token claims |
| batch evaluation → SurrealDB (~30 independently-cloned handles) | Concurrent check_access calls issue overlapping DB round-trips; an unbounded fan-out could exhaust the handle budget |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-27-10 | Tampering / Elevation of Privilege | concurrency-introduced ordering or result-mismatch bug returning a wrong allow/deny | high | mitigate | Mandatory correctness test (batch == per-item results, same order) is an unconditional gate for BOTH transports (Tasks 2-3); `sort_by_key(index)` restores input order |
| T-27-11 | Denial of Service (self-inflicted) | unbounded batch fan-out exhausting the ~30-connection SurrealDB handle budget | high | mitigate | `buffer_unordered(batch_max_concurrency)` with default 16, kept under ~30 (D-07); bound is config-overridable but defaults safe |
| T-27-12 | Elevation of Privilege | a per-item tenant/subject mismatch not rejecting the whole batch (bypass window) | high | mitigate | All identity checks validated synchronously up front; any single mismatch returns permission_denied for the entire batch before any check_access fires — preserves the sequential short-circuit semantics |
| T-27-SC | Tampering (supply chain) | adding `futures` as a direct dependency | low | mitigate | `futures` 0.3.32 is already resolved transitively in Cargo.lock (zero new download); RESEARCH Package Legitimacy Audit verdict OK/Approved (github.com/rust-lang/futures-rs); `cargo add --dry-run` confirms the crates.io source/version before commit |
</threat_model>
