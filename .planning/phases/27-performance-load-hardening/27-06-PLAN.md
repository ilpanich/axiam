<objective>
Extend the CORR-02 `DbManager::reconnect` seam (built forward-compatible for exactly this
work) into a full reconnect loop: full-jitter exponential backoff, a `max_backoff`
ceiling, a bounded retry count, and poisoned-connection eviction via a swappable
`Arc<RwLock<Surreal<Client>>>` handle (D-10/D-11/D-12/D-13). On exhaustion, stay Unhealthy
and keep probing at the ceiling interval forever — never exit the process.

Purpose: Close PERF-04. Full jitter is the AWS-documented fix for reconnect thundering
herds across replicas after a shared DB blip; CORR-03's webhook backoff supplies the
naming/shape but has NO jitter (RESEARCH Pitfall 5) — mirror the naming, ADD the jitter.
Scope is `DbManager`'s own internal handle ONLY — there is no connection pool; do NOT
touch the ~30 pre-existing repository `.client().clone()` snapshots' lifetime model
beyond the mechanical accessor migration (RESEARCH Pitfall 2).
Output: reconnect loop + swappable handle + config knobs + accessor migration + tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DbManager → SurrealDB server | The reconnect loop crosses to the DB over the network; a shared outage can trigger synchronized retries across replicas |
| DbManager handle → repository callers | A poisoned/authenticated-but-broken handle must never be handed to a caller; callers read the current handle via client_cloned |
| reconnect loop → readiness endpoint (/ready via health_check) | Exhaustion must surface as Unhealthy (traffic shed) rather than a masked failure or a crash |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-27-30 | Denial of Service | thundering-herd reconnect storm across replicas after a shared DB blip | high | mitigate | Full-jitter backoff `uniform(0, capped)` (Task 1 `reconnect_backoff_delay`) — the AWS-documented fix; unit test asserts delay ∈ [0, capped] |
| T-27-31 | Integrity / Availability | a poisoned or authenticated-but-broken handle leaked/recycled to callers | high | mitigate | RwLock write-swap drops the old handle atomically; callers only ever read the current handle via `client_cloned`; Task 3 unit test proves the old handle is never returned post-swap (D-12) |
| T-27-32 | Availability | reconnect exhaustion crash-loops or exits the process | high | mitigate | Exhaustion stays Unhealthy and keeps probing at the ceiling interval forever — never break/return/exit (D-11); readiness sheds traffic while self-healing |
| T-27-33 | Repudiation / masked incident | reconnect loop silently swallowing a genuinely revoked root credential as a transient blip | medium | mitigate | The loop reacts to `classify_query_error == DbError::Unhealthy` (CORR-02 D-05) and surfaces it via health_check/`/ready` — it does not swallow the auth-vs-connection classification |
</threat_model>
