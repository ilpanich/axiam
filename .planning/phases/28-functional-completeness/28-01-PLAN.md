<plan_decision id="D-02-input-shape">
`SetOrgEmailConfig.provider` currently carries a non-optional secret (`SmtpConfig.password: String` /
`ApiProviderConfig.api_key: String`), and `set_org_config` re-encrypts unconditionally — so D-02
"omit-preserves" is a real build task here, not a pre-existing behavior. The executor chooses the
mechanism (an input-specific optional-secret representation, or a sentinel-empty convention mapped to
"preserve"), following the federation-config update-secret precedent, but the fixed behavioral contract
is: a write that does not carry a new secret keeps the stored ciphertext; a write that carries a
non-empty secret re-encrypts and replaces it. The response side (D-01) never emits the secret.
</plan_decision>

<plan_decision id="D-13-org-delete-orphaning">
RESEARCH Open Question 2: an org-level DELETE can orphan tenant overrides whose `effective_email_config`
merge depends on an org baseline. We honor D-13 (DELETE exists for both scopes) and add `delete_org_config`.
The orphaning risk is mitigated, not ignored: after an org config is deleted, `get_effective_config`
returns None for its tenants, which D-08's clear "no email config" error surfaces loudly at send time
(expected, actionable admin behavior) rather than silently sending broken mail. No cascade delete of
tenant overrides is performed.
</plan_decision>
