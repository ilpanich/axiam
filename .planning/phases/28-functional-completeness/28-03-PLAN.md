<objective>
Wire per-org/per-tenant custom email templates into the mail send path (FUNC-03 / D-05): thread
`EmailTemplateRepository` into `send_with_retry_and_audit` and `start_mail_consumer`, fetch org and tenant
templates by `msg.org_id`/`msg.tenant_id` + kind, and pass them to the EXISTING
`resolve_template(kind, org, tenant)` (which already implements tenant→org→built-in precedence). Make each
template fetch fail-safe (D-06): on a DB error, log a warning and fall back to the built-in so a broken/unfetchable
custom template can never strand a password-reset or verification email. Do NOT add a template-authoring CRUD
API (deferred) and do NOT modify `resolve_template` itself.

Purpose: Tenants can brand their emails; a custom-template failure degrades gracefully to the built-in.
Output: The consumer renders custom templates when present and the built-in otherwise, always delivering.
</objective>

<plan_decision id="D-06-fetch-only-fallback">
RESEARCH ground-truth: the template renderer (`render`/`render_html`, hand-rolled `{{key}}` substitution) is
INFALLIBLE — it returns a `String`, never a `Result` — so there is no render-error branch to hook. D-06's
fail-safe is therefore implemented ONLY around the two fetch calls (`get_org_template`/`get_tenant_template`,
which return `AxiamResult<Option<...>>` and can genuinely fail). No `catch_unwind` or synthetic render-error
path is added (that would be over-engineering for a function that provably cannot fail today).
</plan_decision>
