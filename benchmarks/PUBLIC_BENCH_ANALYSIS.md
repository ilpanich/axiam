# AXIAM Benchmark Analysis — First Draft

> **Status: first benchmark draft.** This is the initial public run of the AXIAM
> benchmark framework, published to be transparent about both the results *and*
> their limitations. Several cells are invalid (harness bugs, not target
> failures), every figure comes from a single run (not yet the median of ≥3 the
> methodology calls for), and the run was executed on consumer laptop hardware.
> The numbers below are real and reproducible, but they will be refined —
> treat them as a credible first signal, not a final verdict.

## 1. What was measured

Three open-source IAM servers were driven with the **identical logical
workload** through the vendor-neutral k6 harness in
[`benchmarks/`](README.md) (adapter layer isolates per-vendor endpoint
differences; see [`docs/methodology.md`](docs/methodology.md)):

| Target | Server | Datastore | Extra services |
|---|---|---|---|
| **AXIAM** | `axiam-server` 1.0.0-alpha (Rust / Actix-Web / Tonic) | SurrealDB v3 (SurrealKV backend) | RabbitMQ 4 (audit/event broker) |
| **Keycloak** | Keycloak 26.7.0 (JVM) | PostgreSQL 16 | — |
| **Zitadel** | Zitadel v4.15.2 (Go) | PostgreSQL 16 | — |

### Test environment

Everything ran on a single **8-year-old Dell XPS 15 9570 laptop** (12 logical
CPUs, ~31 GiB RAM, Linux + Docker), targets benchmarked sequentially,
back-to-back, never concurrently. Each stack is resource-capped by its
`targets/<name>/docker-compose.yml`:

| Container | CPU cap | Memory cap |
|---|---|---|
| IAM server (all three) | 2 CPUs | 1024 MiB |
| Database (SurrealDB / PostgreSQL) | 2 CPUs | 1024 MiB |
| RabbitMQ (AXIAM only) | 1 CPU | 512 MiB |

Load model (from `scenarios/lib/config.js`): closed-loop **50 VUs**,
ramping-VU executor, **30 s warm-up + 120 s measured + 10 s cool-down** per
scenario, generated by **k6 v2.1.0 running on the host** (outside the caps).
Validity gates: error rate ≤ 1%, p95 latency < 2000 ms.

Two security profiles were run in this round: **p0-plaintext** (HTTP baseline)
and **p2-tls13** (TLS 1.3-only, server auth). All three targets terminate
TLS **in-process** at p2 — AXIAM via its native rustls listener, Keycloak and
Zitadel via their built-in TLS — so the crypto cost lands inside each server's
CPU cap. The p1 (TLS 1.2) and p3 (mTLS) profiles were not exercised in this
run.

**Rate limits:** AXIAM ships a per-IP rate limiter by default (login 10/min,
token 20/min, …). Since k6 drives all load from one source IP, the limiter was
**neutralized** for these runs (`rate_limits: neutralized` in every AXIAM
result's metadata) — otherwise the benchmark would measure the 429 path, not
endpoint capacity. Keycloak and Zitadel ship no equivalent per-IP limit by
default, so this keeps the comparison apples-to-apples.

Resource usage (CPU cores, memory) was sampled every second from
`docker stats` across **all** containers of each stack — so AXIAM's reported
footprint includes SurrealDB **and** RabbitMQ, while competitors' include only
server + PostgreSQL.

## 3. Was the database the bottleneck? Often, yes.

Because resources were sampled per container, we can attribute saturation.
With **2 CPUs / 1024 MiB for the DB** (same as the server), average CPU cores
consumed during the measured window:

| target / scenario (p0-plaintext) | server (cores avg) | DB (cores avg) | bottleneck |
|---|---|---|---|
| AXIAM / authz_check_rest | 0.63 | **2.02 (pegged)** | SurrealDB |
| AXIAM / authz_check_grpc | 0.52 | **1.88** | SurrealDB |
| AXIAM / oauth2_client_credentials | 0.96 | **1.74** | SurrealDB |
| AXIAM / token_refresh | 0.96 | **1.74** | SurrealDB |
| AXIAM / token_introspection | 0.93 | 1.42 | mixed |
| AXIAM / oauth2_password_login | **2.01 (pegged)** | 0.44 | server (Argon2id) |
| AXIAM / jwks_fetch | 1.27 | 0.17 | neither (load generator) |
| Keycloak / *every scenario* | **2.00 (pegged)** | 0.01–0.23 | server (JVM) |
| Zitadel / jwks_fetch | 0.98 | **2.03 (pegged)** | PostgreSQL |
| Zitadel / token_introspection | 1.62 | **2.00 (pegged)** | PostgreSQL |
| Zitadel / userinfo | 0.81 | **2.03 (pegged)** | PostgreSQL |
| Zitadel / oauth2_client_credentials | 1.75 | **1.81** | both near cap |

Three different stories:

- **AXIAM is DB-bound on every DB-heavy flow.** On authorization checks and
  token flows the AXIAM server used 0.5–1.0 of its 2 cores while SurrealDB
  saturated its own 2-core cap. AXIAM's server-side headroom means its
  *server* capacity is higher than these numbers show; the 2-CPU DB cap (and
  untuned SurrealDB/SurrealKV defaults) is what these scenarios measured the
  edge of. Despite that, AXIAM still leads every comparison.
- **Keycloak is server-bound everywhere.** The JVM pinned its 2 cores in every
  single scenario while PostgreSQL stayed nearly idle (≤ 0.23 cores). Giving
  Keycloak's DB more resources would not help it here; giving Keycloak more
  CPU would.
- **Zitadel is mostly DB-bound.** PostgreSQL pegged its cap on
  jwks/introspection/userinfo and ran 1.8+ cores on the token flows, with the
  Zitadel server also near its cap on token issuance — the 2+2 CPU envelope
  constrains it on both sides.

So: yes, the deliberately tight envelope (2 CPUs / 1024 MiB for servers *and*
DBs, mimicking a small-footprint edge/IoT deployment) made the database the
limiting factor for AXIAM and Zitadel on several scenarios, while Keycloak
never got far enough to stress its DB. A follow-up run with uncapped/tuned
databases is needed to measure each *server's* true ceiling (see §6).

## 4. Weaknesses and caveats (the honest section)

**Password login is AXIAM's worst scenario — by design, but too heavily.**
AXIAM hashes with Argon2id at OWASP-recommended parameters. Under 50
concurrent VUs on 2 CPUs it managed only ~35 logins/s with p95 ≈ 2.1 s
(breaching the 2 s validity gate) and drove server memory to ~970 MiB —
consistent with 50 concurrent Argon2id verifications at ~19 MiB each. The
cost of memory-hard hashing is intentional; leaving its concurrency unbounded
is not, and is a known improvement item. Unfortunately this scenario produced
**no valid competitor comparison**: Keycloak's ROPC cells failed 100%
(benchmark seeding issue, not a Keycloak defect) and Zitadel disables the
password grant entirely, so its "login" cell silently falls back to
client-credentials — a much cheaper operation that involves **no password
hashing** — and must not be read as a login comparison.

**userinfo results are invalid for AXIAM and Keycloak.** The harness minted
the bearer token via client-credentials; AXIAM and Keycloak both (correctly)
refuse that service-account token at `/userinfo`, so both cells are 100%
errors — the throughput shown in the raw report for them is the
error-response path and meaningless. Zitadel's cell passed because its seeded
machine user is a real user identity. This is a harness bug; the fix is to use
a user token with `openid` scope for all targets.

**TLS 1.3 costs AXIAM more than it costs competitors on token endpoints.**
AXIAM's native rustls termination is essentially free on introspection
(−0.3%) and JWKS (−13.9%), but halves client-credentials (−55%) and refresh
(−58%) throughput, while Keycloak/Zitadel lose ≤ 5% on the same flows. Notably
neither AXIAM's server nor its DB was CPU-saturated in those p2 runs
(latency roughly doubled instead), which points at a connection/handshake
behavior issue rather than raw crypto cost — under investigation. Even with
the penalty, AXIAM's TLS numbers remain the best of the three.

**Other comparability caveats, stated plainly:**

- Single run per cell; the methodology's median-of-≥3 rule was not yet applied.
- 8-year-old laptop: CPU clock frequency and temperature were not recorded in
  this run. The data argues *against* throttling having distorted the
  comparison — host load stayed moderate (≤ ~4 of 12 threads busy) and
  CPU-bound cells repeated across the 2-hour session agree closely (AXIAM
  introspection p0 vs p2: 2199 vs 2192 req/s ~27 min apart; fully CPU-pegged
  Keycloak client-credentials: 143 vs 138 req/s ~17 min apart) — but
  utilization data cannot distinguish a core at full clock from a throttled
  one, so a *constant* sustained-clock reduction depressing all absolute
  numbers uniformly cannot be excluded. Frequency/temperature telemetry will
  be recorded and published with the next run.
- The resource footprint of AXIAM includes RabbitMQ (~0.1–0.3 cores,
  ~130 MiB) which the competitor stacks don't run — AXIAM's efficiency
  figures are slightly *penalized* by this, not helped.
- k6 verified no server certificates at p2 (throwaway private CA;
  `insecureSkipTLSVerify` — the handshake and record crypto are still real).
- The closed-loop 50-VU model caps measurable throughput on very fast
  endpoints (AXIAM's JWKS number is a floor, not a ceiling).
- gRPC was only benchmarked for AXIAM (Keycloak has no gRPC API; Zitadel's
  gRPC API is planned for a future comparison round).

## 6. Summary and what happens next

**Strengths shown by this first run.** On every scenario that produced a valid
head-to-head, AXIAM leads all three dimensions the framework measures:
throughput (4.4–12× on token issuance, 2.7–6.5× on introspection, 4.3–8.5× on
refresh, 25–48× on JWKS), latency (lowest p50/p95/p99 in every valid
comparison), and resource efficiency (5–12× more requests per CPU core;
lowest CPU-ms per request everywhere) — while performing single-use
refresh-token rotation and doing so with its DB, not its server, as the
saturated component.

**Weaknesses shown by this first run.** Password login breaches the latency
gate under concurrency (unbounded Argon2id needs a concurrency cap); the
authz *batch* endpoints are slower than repeated single checks and the gRPC
batch breaches the p95 gate; TLS 1.3 currently halves token-issuance/refresh
throughput pending a connection-handling investigation; and SurrealDB
