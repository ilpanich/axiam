# Interpreting Results

The report (`results/report.md`, generated by `runner/report.py`) has four
sections. This guide explains how to read them and the traps to avoid.

## 1. All results

One row per (scenario, profile, target). Columns:

| column        | meaning                                                            |
|---------------|-------------------------------------------------------------------|
| `thr(req/s)`  | successful operations per second over the measured window         |
| `p95 / p99`   | end-to-end latency tail (ms)                                       |
| `err`         | failed-iteration fraction; >1% ⇒ row marked invalid (`valid = ✗`) |
| `cpu(cores)`  | avg CPU cores consumed by the **whole target stack** (server+db+mq)|
| `mem(MiB)`    | avg resident memory of the whole stack                            |
| `thr/core`    | **efficiency headline** — req/s per CPU core                       |
| `cpu_ms/req`  | CPU-milliseconds spent per request (lower is better)              |

Always read `thr` together with `p95`. A higher throughput at a much worse p95 is
not a win — fix the load level (`BENCH_VUS`) so latencies are comparable, then
compare throughput and efficiency.

## 2. Efficiency comparison

This is the answer to *"can AXIAM match competitors at a lower resource cost?"*

For each (scenario, profile) with ≥2 targets, it ranks targets by `thr/core`
(🏆 = best) and also shows `thr/GiB` and `cpu_ms/req`. The fair claim to make is:

> "At profile *p2-tls13* on the *client_credentials* scenario, AXIAM sustained
> *X* req/s/core vs Keycloak's *Y* — i.e. ~*X/Y×* the throughput per CPU core at
> equal latency (p95 *a* ms vs *b* ms)."

Do **not** compare across different profiles or different VU levels — only within a
cell. The framework keeps the resource caps equal across targets (see
`docs/methodology.md` §1.3); if you change a target's caps, note it.

## 3. Security cost

For each (target, scenario), the cost of each profile vs the `p0-plaintext`
baseline: `Δ-throughput` (negative = slower) and `Δ-p95(ms)` (positive = more
latency). Use this to:

- Quote what TLS 1.3 / mTLS costs *this implementation* (e.g. "mTLS costs AXIAM
  ~13% throughput, ~5 ms p95").
- Compare *the cost itself* across targets — the implementation that pays the
  smallest penalty for strong security is often the better operational choice even
  if its plaintext number is not the highest.

## 4. Excluded cells

Rows that failed a validity gate (error rate, too few resource samples, k6
threshold breach, health-check failure). They are kept for debugging but never
feed the headline comparisons. If a target dominates an "excluded" section, fix
the run (often: it was overloaded — lower `BENCH_VUS` — or seeding failed) before
trusting any number from it.

## Publishing figures

- Report the **median of ≥3 runs** (`runner/report.py` consumes whatever is in
  `results/`; run the matrix 3× into separate result dirs and median the headline
  numbers).
- Always publish alongside each figure: profile, VU level, resource caps, image
  digests, and host CPU/RAM (all embedded in each `*.meta.json`). A throughput
  number without its envelope is meaningless.
- Never present the AXIAM-only `authz_check_grpc`/`authz_batch_grpc` numbers as
  head-to-head — they have no competitor equivalent and are labelled non-comparative.
